## §5 五类失败回滚

### 场景一：SSH/部署版本不一致

**症状**：目标机分支、commit 或 dist 与主仓库不一致。  
**路径**：停止后续脚本；保存 `git status`、`git log` 和部署日志；回到上一已验证 commit，重新同步后再部署。  
**RTO**：15 分钟。

### 场景二：数据库迁移或 alembic 多 head

**症状**：upgrade 失败、表/列缺失或出现 multiple heads。  
**路径**：不强行 `upgrade heads`；恢复数据库备份，核对 migration down_revision，按单链修复并在 staging 重跑。  
**RTO**：30 分钟。

### 场景三：VAPID、Service Worker 或 PWA 回归

**症状**：推送密钥变化、manifest 410、SW cache 污染或页面白屏。  
**路径**：保留旧 dist/SW；恢复上一版本；清理客户端缓存仅作为辅助；重新用 `npm run build` 生成 hashed manifest。  
**RTO**：20 分钟。

### 场景四：qa-bench gate 或 Desktop E2E 失败

**症状**：D5 低于 90、per-intent 退化、核心页面或聊天失败。  
**路径**：标记部署为未通过；保留失败样本和日志；回滚应用版本，不扩大用户流量；由主指挥决定修复或暂停。  
**RTO**：30 分钟。

### 场景五：清理脚本误删或数据完整性异常

**症状**：cleanup apply 影响非目标对象、Drive/聊天数据缺失或权限异常。  
**路径**：立即停止 cleanup/celery；从备份恢复受影响数据；核对 dry-run 范围、租户/用户过滤和审计日志；未完成复核前禁止重跑。  
**RTO**：60 分钟，数据恢复时间另计。
