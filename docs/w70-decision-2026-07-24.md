### 1.3 五个 hot-fix 的意义

5 个 hot-fix 并非新增产品范围，而是将“可以合并”提升为“可以部署验证”。它们主要解决以下类别：

- CI 或脚本参数在真实 shell 环境中的兼容性；
- qa-bench 真跑入口的可操作性；
- VAPID setup 与 verify 的持久化闭环；
- 部署 verify 与清理脚本的安全边界；
- 文档、memory 和批次状态之间的引用一致性。

因此，本批的完成标准应以**可执行证据**而非 commit 数量衡量。15 agents 和 5 hot-fix 提供了候选交付面，主指挥部署后的 8 步 checklist 才是最终验收。

### 1.4 路线 B/C/D/E 五路线实跑结论

**路线 B：Drive v2。** 现有能力已经进入连续迁移和部署阶段。继续派工前，必须先确认 W68 第 5—10 批在云 server 的迁移链能够单链升级，避免本地代码状态与生产数据库状态脱节。

**路线 C：Desktop/Mobile。** UI 侧的主要风险从“组件缺失”转为“真实浏览器和真实后端联通”。因此 Desktop v3.2 e2e 是必要的独立证据，不应以静态构建成功替代。

**路线 D：qa-bench。** 评测脚本已具备 1000 题真跑条件，但在真实运行前，任何 gate 结论都只能视为预测。Phase 2 真跑结果应成为 W69 派工与 Q4 路线排序的输入。

**路线 E：部署、文档、守恒。** 文档、memory、验证脚本和清理脚本已经形成主指挥执行面。其价值在于让部署动作可复盘、失败可回滚、完成可证明。

### 1.5 批次完成判定

W68 第 10 批可判定为“开发侧完成、部署侧待验收”。这不是未完成，而是有意将最后的质量闸门留给真实环境。判定条件如下：

- 代码与文档已经进入 main；

## §3 主指挥决策 checklist

预计总耗时 **90—150 分钟**。每一步应记录开始时间、结束时间、命令、输出摘要和是否通过。任何失败都进入 §5，不允许跳过失败继续宣称全量通过。

### 3.1 八步清单

1. **部署 W68 第 5+7+8+9+10 批**  
   主指挥通过 SSH 将主仓库 main（含 15+4 commits）部署到云 server。部署前确认工作树、镜像版本、迁移文件和配置快照；部署后重启必要服务并保存 commit hash。

2. **初始化 VAPID 持久化**  
   ```bash
   bash scripts/setup_vapid_persistence.sh
   ```
   该步骤只执行一次。记录生成/迁移的密钥位置、权限、容器挂载关系和脚本退出码。若脚本提示已有持久化配置，不要覆盖原密钥。

3. **验证 W68 第 7 批部署**  
   ```bash
   bash scripts/verify_w68_7th_batch_deployment.sh
   ```
   记录 API、服务健康、关键配置和响应状态。验证失败时先保存完整日志，再进行回滚或针对性修复。

4. **清理已完成 worktree 和分支**  
   ```bash
   bash scripts/w68_7th_batch_cleanup_plan.sh --apply
   ```
   预期删除 15 个 worktree、16 个分支。执行前应确认没有未合并 commit、未保存改动和仍需取证的 agent 分支；执行后重新运行 git worktree/branch 列表核对。

5. **真实运行 Phase 2：1000 题**  
   ```bash
   bash tests/qa-bench/run_d5_dry.py --full --per-intent --gate-threshold 90
   ```
   保存完整输出和结果文件。重点记录总体通过率、per-intent 分布、gate 是否达到 90、超时/工具调用/格式错误分类，以及与历史 baseline 的差异。

6. **验证 VAPID 重启持久化**  
   ```bash
   bash scripts/setup_vapid_persistence.sh --verify
   ```
   verify 必须确认重启或重新加载后公钥/私钥仍一致，且应用仍能读取。不得仅以文件存在作为通过条件。

7. **执行 Desktop v3.2 端到端 e2e**  
   由主指挥手动执行，或派 1 个 agent 专门执行，不扩大 W69 派工。覆盖登录、核心页面、评论/分享路径、主题样式和关键 API 联通；保存截图、控制台错误和失败步骤。

8. **验证 baseline 守恒**  
   ```bash
   bash tests/test_baseline_audit.py
   ```
   预期 **71 PASS + 7 SKIP**。任何 PASS 减少、SKIP 增加、测试收集错误或新增未解释失败，都视为未通过，需要进入失败记录。

## §4 长期路线：Q4 2026 商业化打包

W70 不建议在验收前启动大规模商业化开发，但应提前形成资源量级和优先级视图。以下估算为人月，实际排期需在产品、客户、合规和基础设施验证后确认。

### 4.1 Phase 8：实时语音科研助手（4 人月）

目标是将当前录音机 + 离线后处理能力扩展为低延迟实时语音科研助手。范围包括实时 ASR、增量说话人识别、实时摘要提示、实验上下文检索和可控的 TTS 反馈。必须延续现有跨 event loop、流式 abort、原始转录保护和会议标准格式纪律。

验收重点：端到端延迟、断网恢复、GPU 资源隔离、声纹误认率、会议记录完整性和人工关闭实时能力的降级路径。

### 4.2 多组织 SaaS（4 人月）

目标是从单课题组部署演进为多组织隔离的 SaaS 能力。范围包括 organization/tenant 模型、成员与角色边界、配额、账单准备、组织级知识库和审计日志。

验收重点：所有查询的 tenant 过滤、跨组织越权测试、数据导出/删除、密钥隔离、限流、备份恢复和管理员审计。

### 4.3 桌面 EXE 打包（4 人月）

目标是提供面向实验室本地电脑的桌面安装包，降低 Docker、GPU Whisper、FRP 和本地服务的部署门槛。范围包括安装器、升级器、配置向导、服务健康诊断和卸载清理。

验收重点：Windows 兼容、升级回滚、GPU/CPU fallback、日志导出、网络隧道诊断、数据目录保护和签名/发布流程。

### 4.4 多平台原生 APP（6 人月）

目标是将移动端核心能力扩展为稳定的 iOS/Android 原生或跨平台应用。范围包括推送、离线缓存、录音、分享、生物识别、后台任务和应用商店合规。

验收重点：iOS/Android 生命周期、离线同步冲突、权限最小化、端到端加密、推送密钥轮换、深色模式、无障碍和应用商店审核。

### 4.5 资源总览与排序

四条路线合计约 **18 人月**（4 + 4 + 4 + 6）。若考虑并行协调、测试、产品和上线缓冲，商业化打包可按约 **24 人月**的整体资源池预留，而不是把 18 人月视为可直接承诺的工期。

建议 Q4 排序：

1. 多组织 SaaS：决定商业化数据边界和收费基础；
2. Phase 8 实时语音：强化科研场景差异化；
3. 桌面 EXE：降低部署成本和支持成本；
4. 多平台原生 APP：在推送、离线和组织模型稳定后扩大触达。

排序不是最终承诺。应以 §3 的真实数据、用户访谈和 Q4 资源确认进行季度拍板。

## §5 失败回滚

### 5.1 回滚总原则

任一步失败都应先保留证据，再停止扩散；不得为了“完成 checklist”而跳过红色结果。回滚目标是恢复 W68 部署前的稳定版本，不是临时删除数据或强行修改生产数据库。

### 5.2 五类失败回滚路径

1. **部署或服务启动失败**  
   停止后续验证，保留 SSH、容器、迁移和健康检查日志；恢复上一稳定镜像/commit，回滚未完成迁移，重启 app、worker 和相关服务；确认首页、登录和核心 API 恢复后再决定是否重试。

2. **数据库迁移或 alembic 链失败**  
   立即停止升级，不执行 `upgrade heads` 绕过问题。记录当前 revision、heads、迁移文件和数据库备份；恢复到部署前 revision，修正单链顺序后在 staging 重放，再安排新的窗口。

3. **VAPID setup/verify 失败**  
   不覆盖旧密钥，不删除现有持久化目录。恢复原挂载和环境变量，重启服务验证旧公钥仍可读取；若新密钥已被部分写入，先隔离临时文件，再由主指挥决定是否重新初始化。

4. **Phase 2 1000 题或 Desktop e2e 失败**  
   保存原始报告、失败题目、浏览器日志和网络响应；停止新增派工。若属于环境配置，修正并小范围重试；若属于代码回归，回滚到 W68 部署前版本，并将可复现根因列入 B 选项候选。

5. **Baseline audit 守恒失败或清理误删风险**  
   立即停止清理和后续发布。若 worktree/分支尚未 apply，取消 cleanup；若已 apply，依赖 git reflog、远端分支和已保存 commit hash 恢复。对 baseline 回归执行 `git bisect` 或按批次比较，不以删除测试来恢复数字。

### 5.3 回滚后的重新决策

回滚成功后不自动进入 W69。主指挥应先完成三项复盘：失败证据归档、根因分类（环境/数据/代码/流程）、是否能在单一小修中闭环。只有根因明确且收益高，才转选项 B；若问题反映部署窗口或资源不足，则转选项 D；选项 C 必须重新获得独立的真实数据支持。
