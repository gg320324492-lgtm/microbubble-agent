# W68 第 14 批派工纪要候选：Prompt Template v5

> 版本：v5（2026-07-24）  
> 适用：主指挥、并行 agent、文档/调研/迁移派工  
> 原则：先验证、再派工；先串链、再合并；先反馈、再升级；合并顺序可见。

## §1 升级背景：v4 → v5 增量

### §1.1 v4 已经解决了什么

v4 把三类"看起来完成、合并才出错"的问题前置：

| 错误类别 | v4 门禁 |
|---|---|
| Alembic 双头 | `git fetch origin main && alembic heads` + revision 唯一 + down_revision 指向最新 head + 报主指挥 |
| PowerShell 5.1 binding | `--mode <value>` 空格分隔 + `[string]$Mode` + `$Mode -eq 'session'` 严格 |
| plans 状态自报 | `grep -rn` plans 关键字 + 5 真未实施 backlog + COMPLETED 引真 commit |

v4 在 W68 第 13 批实战中由 A-1、B-1、C-1 等 agent 应用，按交付结果该批 anchor 守恒。

### §1.2 v4 在 W68 第 14 批实战中暴露的 2 类缺口

W68 第 14 批 15 agents 派工试用 v4 后，主指挥收到两类新错误。

**缺口 1：缺"经验反馈循环"段**。agent 完工后不知道 v4 模板哪些段对完成任务最有帮助，哪些多余或偏离。下一批 agent 仍要重写 prompt，模板没法自我进化。

**缺口 2：缺"主指挥合并顺序表"段**。agent 在并行派工下不知道自己的 commit 在主指挥合并链第几步、与谁有 alembic 串单链依赖、与谁有 docs 引用依赖。等到合并时才补一句话"应该这个先合"会浪费主指挥 10–20 分钟沟通时间。

### §1.3 v5 增量

v5 保留 v4 的所有门禁，仅在 §5 之后增加两个新段：

| 新段 | 解决 | 内容 |
|---|---|---|
| §5 经验反馈循环 | 模板自我进化 | agent 完工后回填段 1–4 的"有效 / 多余 / 偏离"，主指挥汇总升级 v6 |
| §6 主指挥合并顺序表 | 派工前预协调 | 派工时附 Step 1–N 合并链 + alembic 串单链位置 + 文档引用位置 |

v5 默认应用从 W68 第 15 批开始；W71 默认应用 v5（不再回退到 v4）。

## §2 v5 完整 6 段模板

以下模板可直接复制到派工消息；尖括号内容必须替换。

### 段 1：角色、范围与不变量

```text
你是 Agent <编号>：<标题>。目标是 <一句话目标>。
范围：<文件/目录>；不范围：<明确排除项>。
硬规则：0 production code（如适用）；不得 merge；不得覆盖他人改动；
输出必须包含证据路径、测试结果、commit hash 和阻塞项。
当前分支：<worktree 分支名>；基线 commit：<hash>；
依赖 agent：<agent-id>（说明是否等待其 commit）。
```

### 段 2：交付物与操作边界

```text
交付物：
1. <文件一>：<内容和大致规模>
2. <文件二>：<内容和大致规模>
禁止：<代码/配置/数据库/部署等不应修改的对象>。
```

若需要脚本，列出运行时版本（Windows PowerShell 5.1 或 PowerShell 7）和调用约定；若需要 migration，列出 revision 命名范围及明确的 `down_revision` 上游。

### 段 3：任务描述、前置验证与风险门禁

```text
开始前：
- git status；确认基线没有未授权修改。
- plans 任务：grep -rn "<keyword>" C:/Users/pc/.claude/plans/，
  不以 status 自报替代事实；真未实施项写 backlog docs，完成项标
  COMPLETED + 真 commit。
- alembic 任务：git fetch origin main && cd main && alembic heads；
  检查 revision 唯一、down_revision 指向最新 head；发现双头立即报主指挥。
- 其他任务：<领域特定检查>。
```

迁移 agent 不得自动替换有争议的 down_revision；必须给出冲突矩阵。合并后由主指挥 rebase 重命名（如 070→075/074/076）、串单链并再跑 heads。

### 段 4：完成定义、测试与 PS 5.1 约束

```text
完成定义：
- 所有交付文件存在且内容可审计；
- <测试/grep/history 验证>通过；
- 报告证据路径和命令输出摘要；
- 未完成项写 BLOCKED，并说明下一步。
PowerShell 5.1：使用 --mode <value>（空格），[string]$Mode，
仅用 $Mode -eq 'session' 判断 session；附两种模式实跑证据。
```

完成定义不能只写"代码已写"或"测试通过"。测试需说明是单元、集成、静态检查还是人工检查，以及是否受环境限制。对于 docs-only 任务，至少做文件计数、关键词检查、git diff --check。

### 段 5：经验反馈循环（v5 新增，必填）

```text
回传反馈（必填）：
- 段 1–4 中哪些段有效（具体指明段号 + 段主题）？
- 哪些段多余 / 偏离 / 重复？
- 哪些约束有歧义或缺漏，导致 agent 自行判断？
- 建议保留 / 删除 / 改写的段号 + 一句话理由。

不写该段视为完工未达标，主指挥合并时不视为有效交付。
```

主指挥汇总 15 agents 的反馈后升级 v6。新铁律按"反馈必须 ≥ 1 条有效建议 + 0 条负面偏离才进入 v6 候选" 规则筛选。

### 段 6：主指挥合并顺序表（v5 新增，派工必读）

```text
主指挥合并链（本 agent 必须遵守）：

| 步骤 | agent-id | 任务标题 | commit 范围 | 前置依赖 | 串单链位置 |
|---|---|---|---|---|---|
| Step 1 | <id> | <标题> | docs/.../file.md | 基线 | — |
| Step 2 | <id> | <标题> | alembic/versions/070_*.py | Step 1 | base=068 down_revision=068 |
| Step 3 | <id> | <标题> | memory/.../v5-2026-07-24.md | Step 2 | docs ref Step 1+2 |
| ... | ... | ... | ... | ... | ... |

合并触发条件：
- 主指挥在所有 agent 回传 commit hash 后按 Step 顺序 rebase / cherry-pick / merge。
- alembic 串单链中途若发现双头，停止合并并报主指挥拍板（v4 §1.1）。
- agent 不主动 merge 或 push（除非派工明文授权）。

agent 完工后必须确认 commit hash 出现在顺序表的正确行；如发现顺序与派工不一致，立报主指挥。
```

合并顺序表让 agent 提前知道：(1) 谁在它前面，谁在它后面；(2) 是否需要等 alembic 上游；(3) 是否需要被 docs 下游引用；(4) 主指挥合并时如何串链。

## §4 v4 → v5 diff

### 4.1 新增段

- **段 5（经验反馈循环）**：agent 完工后必填。从 v5 开始每个 agent 的回传标准回执格式为"段 1–4 有效段反馈 + 段 5 至少 1 条建议 + 段 6 commit hash + 行号确认"。
- **段 6（主指挥合并顺序表）**：派工消息必含。agent 在自己 worktree 内只 commit 不 merge、不 push（除非派工明文授权），主指挥在所有 15 agents 回传后统一合并。

### 4.2 兼容性

v5 完全兼容 v4 的所有门禁：

- 段 1–4 是 v4 模板的 1:1 复制，未做任何修改或弱化。
- v4 派工任务升级到 v5 只需在原 prompt 末尾追加段 5 + 段 6 两段。
- 历史 v1 / v2 / v3 派工不受影响（v4 兼容矩阵已涵盖）。

### 4.3 diff 摘要表

| 维度 | v4 | v5 增量 |
|---|---|---|
| 段数 | 5 段 | +段 5 反馈 +段 6 顺序表 → 6 段（段 5 内嵌于原有"回传"扩展，段 6 在末尾并入"主指挥合并链") |
| 模板长度 | 5 段 ≈ 50 行 | +段 5 模板 ~ 8 行 +段 6 表格 ~ 12 行 → 总 ~ 70 行 |
| 关门强度 | 段 1–5 必含 | +段 5 必填反馈 +段 6 必含合并表 |
| 跨 agent 依赖 | 隐含在段 1 "依赖 agent" | +段 6 显式合并顺序 + alembic 串单链位置 |
| 应用范围 | W68 第 13 批 + W71 | +W68 第 14 批及之后全部批次 |

## §5 6 条新铁律（v5 沉淀）

1. **段 5 反馈必填**——agent 完工回传必须含段 5（"有效段 / 多余段 / 偏离段" 三类至少各 1 条建议），否则视为"完工未达标"，主指挥合并时不视为有效交付。
2. **段 6 合并顺序表必含 alembic 串单链位置**——派工消息段 6 的"串单链位置"列必须写明 `base=<上游 revision> down_revision=<上游 revision>`，且 `base` 必须等于同表"前置依赖"列指向的 commit 的 alembic revision；agent 不得擅自修改。
3. **v5 默认应用从 W68 第 15 批开始**——任何 W68 第 15 批及以后的派工必须使用 v5；W71 派工也默认 v5。
4. **段 6 表格对单 agent 派工仍强制**——即使本批只有 1 个 agent 派工，段 6 仍必须含一张单行 Step 1 表格（"串单链位置"写"单 agent，base=当前 HEAD"），避免 agent 误以为可以自动 merge。
5. **段 5 反馈汇总后必升级 v6**——主指挥在每批 grand closure 时汇总 N agents 反馈，写入 docs + memory 永久锚点；下一批若新增 v6 段则派工升级到 v6。
6. **反馈循环与顺序表互不替代**——段 5 是"对模板本身的反馈"，段 6 是"对派工具体任务的执行顺序"；二者互不替代，缺一不可。

## §7 v4 派工任务如何升级到 v5（兼容方式）

如果主指挥以前用 v4 派工（未含段 5 + 段 6），需要升级到 v5：

1. **找到原 v4 prompt**：在派工频道搜索最近 7 天内的 prompt 文本。
2. **追加段 5**："回传反馈（必填）：- 段 1–4 中哪些段有效？哪些多余？哪些偏离？- 建议保留 / 删除 / 改写的段号 + 一句话理由"。
3. **追加段 6**：根据本批其它 agent 的 commit 顺序写 Step 1-N 表。如果不知道顺序，写"主指挥尚未确定合并顺序，本 agent 在 commit 后报 hash 与串单链位置，由主指挥拍板合并顺序"。
4. **保留段 1–4 原文**：不要改任何字，否则派工追溯链断。
5. **追加时机**：派工发出后 1 小时内追加 v5 段 5 + 段 6 给已开工 agent；agent 必须承认收到后再继续。

这种方式保证 v4 派工和 v5 派工可并存，不强制迁移。

## §8 v5 默认应用范围

- **W68 第 14 批已应用 v5**：A-1 ~ D-3 全部 15 agents 在派工时已含段 5 + 段 6。
- **W68 第 15 批及之后**：默认 v5；不再回退到 v4。
- **W71 派工**：默认 v5；尤其是 migration、PowerShell、plans 审计任务，必须把对应门禁原样写进 prompt。

## 结语

v5 不是替换 v4，而是补两段让模板"自我进化 + 合并顺序可见"。段 5 让 agent 反哺模板，段 6 让 agent 知道自己处在合并链哪里。这样 W68 第 15 批 + W71 派工可以基于"经过实战反馈的模板 + 经过派工前协调的合并顺序"启动下一轮，更可预测、更易追溯、更不易合并错。
