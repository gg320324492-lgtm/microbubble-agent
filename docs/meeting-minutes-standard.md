# 会议纪要标准格式

> 生效时间：2026-06-06  
> 适用范围：所有会议 AI 分析、手动优化会议内容、历史会议补写、对话中要求“优化会议内容”的场景。

## 总原则

会议纪要必须按 `2026.5.28 例行例会` 的信息密度和结构输出，不能只写短摘要。优化时要保留真实转录中的事实，不得虚构人物、结论、任务或实验数据。

## 标准结构

### 1. 标题

标题应概括会议核心主题，优先使用“主题 + 会议类型/动作”的形式。

示例：
- `小气助手软件适配性测试`
- `臭氧微纳米气泡实验异常讨论`
- `2026.5.28 例行例会`

### 2. 摘要

摘要使用一段完整文字，长度通常为 3-6 句。必须覆盖：
- 会议围绕什么问题展开
- 谁提出或演示了什么
- 讨论过程中的关键分歧、测试目标或背景
- 最后形成了什么共识、下一步方向或待补信息

摘要不能只写“会议讨论了 X，并决定 Y”。要写出背景、过程、结论和后续动作。

### 3. 讨论要点

`key_points` 必须是数组，每条都使用：

```text
【发言人】具体内容
```

要求：
- 每条保留一个明确观点或事实，不要把多个主题硬塞进一条。
- 同一发言人重复表达的相同主题合并为一条。
- 对重要演示、实验观察、参数、问题、软件模块、硬件配置要求等要展开到可理解程度。
- 如果发言者无法确认，使用 `【发言人A】` / `【发言人B】`，不要强行猜姓名。

### 4. 决议事项

`decisions` 必须是数组，每条都使用：

```text
【发言人/双方/全组】决定或共识内容
```

要求：
- 写清楚谁决定、决定做什么、目的是什么。
- 对“继续测试”“整理清单”“记录参数”等事项要说明后续用途。
- 没有明确责任人时使用 `【双方】` 或 `【全组】`，不要凭空指派。

### 5. 待办任务

只有在转录中出现明确行动项时才生成 `action_items`。每个任务需要包含：
- `title`
- `assignee_name`
- `description`
- `due_date`
- `priority`
- `confidence`

不明确的行动项不要过度推断。

## 优化示例

不合格：

```text
会议围绕小气助手软件适配性展开，决定继续测试并整理硬件信息。
```

合格：

```text
会议围绕“小气助手”软件的实际适配性测试展开。杜同贺首先演示了系统首页和多个核心功能模块，包括仪表盘、任务管理、会议管理、项目管理、成员管理、知识库和长期记忆等，并结合当前系统中已有的会议记录、轻量服务器、微纳米气泡课题组、智能 Agent 和 ChatGPT 等相关内容，说明该系统已经具备较完整的课题组日常管理与科研协作支撑能力。吴孟铨随后提出本次测试的重点是验证软件在当前使用环境下是否适用、是否适配，并表示将继续进行测试。
```

## 维护要求

- 手动修改数据库会议内容时，必须同时检查 `summary`、`key_points`、`decisions` 和 `transcript_polished`。
- 自动分析 prompt 修改时，必须同步更新本规范、`README.md`、`ROADMAP.md`、`CLAUDE.md`、`AGENTS.md` 中的相关说明。
- `transcript` 原始转录不改；只优化 `transcript_polished` 和结构化纪要字段。
