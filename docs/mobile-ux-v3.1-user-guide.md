## 1. 语音输入 (G-1)

语音输入是 v3.1 的核心移动端体验升级 — 在 `MobileChatView` / `MobileCommandPalette` / `MobileSearchSheet` 三个高频输入场景默认开启, 让"边走边说"成为可能。

### 1.1 三态交互 (按住 / 上滑 / 松开)

| 状态 | 触发动作 | 视觉反馈 | 触觉反馈 |
|------|---------|---------|---------|
| **空闲** (idle) | 看到 🎤 按钮 (在输入框右侧) | 灰色实心话筒 | 无 |
| **录音中** (recording) | **长按** 🎤 0.3s 触发 | 红色脉冲圈 + 实时波形 + 计时器 | `tap` (10ms 短震) |
| **取消中** (cancelling) | 按住时**向上滑**超过 80px | "松手取消"气泡 + 录音图标淡灰 | `warning` `[30, 50, 30]` |
| **发送** (sending) | 松开手指 (未上滑取消) | 波形收尾 + 自动转文字 | `success` `[10, 50, 10]` |

>
> 录音三态基于**长按手势 + 垂直位移**双维度判定, 完整交互描述如下:
>
> **状态机表**:
>
> | 状态阶段 | 触发条件 | 视觉表现 | 触觉反馈 |
> |---------|---------|---------|---------|
> | **idle** (默认) | 看到 🎤 按钮 (输入框右侧) | 灰色实心话筒图标 | 无 |
> | **holding** (预热) | **长按 0.3s** 触发 | 图标变红, 系统申请麦克风权限 (首次弹原生框), 计时器开始 | `tap` (10ms 短震) |
> | **recording** (录音中) | 权限授予后 | 红色脉冲圈 + 实时波形 (▁▃▅▇▅▃ 滚动) + 计时器 mm:ss | 无 |
> | **cancelling** (取消中) | 按住时**向上滑** ≥ 50px | "松手取消" 气泡浮现, 录音图标淡灰, 波形停止更新 | `warning` `[30, 50, 30]` |
> | **sending** (发送中) | 正常松手 (位移 < 50px) | 波形收尾 → ASR 服务调用 → 转写文字填入 | `success` `[10, 50, 10]` |
>
> **使用流程**:
>
> 1. 用户 **长按 🎤 按钮** 0.3 秒
> 2. (首次) 系统弹麦克风权限框 → 用户点 **"允许"**
> 3. 图标变红 + 脉冲圈出现 + 实时波形开始滚动 (每秒约 5-8 个波峰)
> 4. (可选) 如果说错, **不要松手**, **按住向上滑 ≥ 50px** 进入取消区, 出现 "松手取消" 气泡
> 5. 松手: 取消状态则丢弃录音; 录音状态则上传 + ASR 转写
> 6. 转写文字自动填入输入框 (不直接发送, 可编辑)
>
> **取消阈值细节**: 50px 是黄金值 — 太小 (如 20px) 容易误触取消; 太大 (如 100px) 用户要滑很远才能取消, 体验差。50px 拇指自然上滑距离的临界点。

### 1.2 详细操作步骤

#### 步骤 1: 长按开始录音
1. 打开小气助手 → 进入 `MobileChatView` (或任何带 🎤 的页面)
2. 找到输入框右侧的 🎤 麦克风图标
3. **长按** 约 0.3 秒 — 会感受到一次轻微震动 (`tap`)
4. 松开长按前的"等待"期: 系统正在请求麦克风权限 (首次会弹原生权限框)
5. 权限授予后: 录音正式开始, 图标变红 + 实时波形显示

#### 步骤 2 (可选): 上滑取消
- 如果说错话了, 不要松开, **继续按住并向上滑动** 超过 80px
- 输入框上方出现 "松手取消" 气泡
- 此时松开手指, 录音被丢弃 (不发送, 不入 ASR 队列)
- **常见误区**: 有些人会习惯性下滑试图"撤回", 这在我们的设计里**没有效果** — 只有上滑才算取消

#### 步骤 3: 松开发送
- 正常说完后, **直接松开手指** (不要上滑)
- 系统自动停止录音 + 上传到 ASR 服务
- ASR 转写完成后, 文字自动填入输入框 (不直接发送, 你可以再编辑/补字)
- 文字出现时伴随一次 `success` 触觉反馈, 确认转写成功

#### 步骤 4 (可选): 二次编辑
- 转写文字会**自动填入**输入框但**不发送**, 你可以:
  - 直接按 ➤ 发送
  - 手动修改错字 (ASR 在中英混说 / 专业术语 / 噪声环境下会有 5-10% 错误率)
  - 继续按 🎤 追加录音 (最长单次 60s, 超过会自动分段)

### 1.3 iOS Safari 权限流程

iOS Safari 对麦克风权限管控最严格, 首次录音会经历 3 步:

| 步骤 | 触发 | UI 表现 | 用户操作 |
|------|------|---------|---------|
| 1 | 长按 🎤 | 浏览器原生权限弹窗: "允许 'xxx.xx' 使用麦克风?" | 点 "允许" / "不允许" |
| 2 | 允许后 | 状态栏顶部出现 🎙️ 红色图标 (系统级, 不可关闭) | 无需操作, 表示正在录音 |
| 3 | 首次以后 | 直接进入录音中状态 (无弹窗) | 无需操作 |

> **⚠️ iOS Safari 14+ 特别说明**:
> - 在 **非 PWA (浏览器 tab)** 模式下, 每次进入页面都可能需要重新授权 (iOS 14.5+ 隐私策略)
> - 在 **PWA standalone 模式** 下, 授权会**持久化** — 一次授权, 长期有效 (添加到主屏后)
> - 如果误点了 "不允许", 需要去 `设置 → Safari → 高级 → 网站设置 → 麦克风 → 找到本站 → 允许`
> - **HTTPS 强约束**: 麦克风 API 只在 HTTPS 下可用, HTTP 站会直接拒绝 — 我们的服务都部署在 HTTPS + Nginx 反代后

### 1.4 录音时长限制

| 限制 | 数值 | 原因 |
|------|------|------|
| 单次最长 | 60 秒 | 移动端 ASR 服务端单请求 60s 硬上限 (faster-whisper / SenseVoice batch 上限) |
| 单次最短 | 0.5 秒 | 低于此时长视为误触, 直接丢弃 |
| 每日上限 | 无 | 走 Celery 队列, 不限频次 (但高峰时段排队可能 5-10s) |
| 文件大小 | ~500KB / 60s | Opus codec @ 32kbps, 移动端 4G 网络下 < 1s 上传完成 |

> 如果你需要录制超过 60 秒的长内容 (例如会议录音), 请使用 **录音机** 功能 (顶部导航 → 会议 → 录音机), 该功能是离线分段 + 后台异步处理模式, 不受 60s 限制。

### 1.5 ASR 转写引擎选择

v3.1 默认使用 **SenseVoice** (阿里达摩院开源, 2026-06-30 benchmark 5/5 胜 faster-whisper), 自动按以下优先级降级:

```
SenseVoice (主) → faster-whisper (备) → Paraformer (兜底) → 本地缓存错例
```

中文 / 中英混说 / 噪声环境表现最佳; 纯英文 / 其他语种自动降级到 faster-whisper。

**ASR 服务完整调用链**:

```
┌─────────────────────────────────────────────────────────────────────────┐
│  前端 (MobileChatView)                                                   │
│  ↓                                                                     │
│  MediaRecorder.start() → 采集 audio/webm (Android) / audio/mp4 (iOS)    │
│  ↓                                                                     │
│  MediaRecorder.stop() → 产生 Blob (~500KB / 60s)                       │
│  ↓                                                                     │
│  POST /api/v1/voice/asr                                                │
│  Body: { audio: <Blob>, mime: "audio/webm", duration_ms: 12345 }      │
│  Headers: Authorization: Bearer <JWT>                                  │
│  ↓                                                                     │
│  ASR Router (settings.VOICE_ASR_BACKEND=sensevoice)                    │
│  ↓ 优先: SenseVoice → 失败重试 1 次                                     │
│  ↓ 降级: faster-whisper → 失败重试 1 次                                  │
│  ↓ 兜底: Paraformer → 失败抛 503                                        │
│  ↓                                                                     │
│  STT 转写 → 返回 { text: "你好", confidence: 0.93, language: "zh" }   │
│  ↓                                                                     │
│  前端填入输入框 (is_partial=false, 不直接发送)                            │
└─────────────────────────────────────────────────────────────────────────┘
```

**引擎对比表**:

| 引擎 | 语言优势 | 速度 | 错误率 | 何时用 |
|------|---------|------|--------|--------|
| **SenseVoice** | 中文 / 中英混说 | ~0.5s (60s 音频) | 3-5% | 主选, 默认 |
| **faster-whisper** | 多语种, 英文 | ~1.5s (60s 音频) | 5-8% | 纯英文 / SenseVoice 失败时 |
| **Paraformer** | 中文 | ~0.8s (60s 音频) | 5-7% | 兜底, GPU 紧张时 |

> **用户手动选择入口**: 暂未提供 UI 切换, 但在 **设置 → 高级 → ASR 引擎** (debug 模式) 可强制指定。普通用户无需关心, 自动降级策略已覆盖 99% 场景。

### 1.6 错误处理与降级

| 错误场景 | UI 表现 | 降级方案 |
|---------|---------|---------|
| 麦克风权限被拒 | "请允许麦克风权限" 提示 + 设置引导链接 | 引导去系统设置 |
| 浏览器不支持 MediaRecorder | 🎤 按钮变灰 + tooltip "当前浏览器不支持" | 仅企业微信 X5 / HarmonyOS ArkWeb 等极老内核, 普通用户遇不到 |
| getUserMedia 5s 超时 | "无法访问麦克风, 请检查系统权限" | HarmonyOS / 企业微信已知问题, 引导刷新或换浏览器 |
| ASR 服务 5xx | "语音转文字失败, 请重试或手动输入" | 自动重试 1 次 + 保留本地音频, 可手动触发重试 |
| 网络断网 | 录音正常完成, 排队到 IndexedDB, 恢复后自动重试 | v3.0 IndexedDB 队列兜底 |

## 2. 手势导航 (G-2)

v3.1 在 v3.0 长按菜单 + 响应式断点的基础上, 进一步引入 iOS 风格的"滑切"手势, 让单手操作成为可能。

### 2.1 左右滑切换页面

> **核心规则**: 在 **Tab 栏同级** 的 4 个主页面 (Dashboard / Task / Knowledge / Workspace) 之间左右滑切换, 不影响子页面栈。

| 起始页 | 左滑 → | 右滑 → | 备注 |
|--------|--------|--------|------|
| Dashboard | Task | (无) | 第 1 页, 右滑无效 |
| Task | Knowledge | Dashboard |  |
| Knowledge | Workspace | Task |  |
| Workspace | (无) | Knowledge | 末页, 左滑无效 |

#### 触发条件
- **横向位移** ≥ 50px
- **滑动时长** < 300ms (避免长按后拖拽被误判)
- **起始位置** 距离屏幕左右边缘 ≥ 20px (避免误触系统级右滑返回)
- **页面栈深度 = 1** (根页面), 子页面 (例如点入 Task 详情) 不可滑

#### 视觉反馈
- 滑到 ≥ 50px 但未松手: 整个页面向左/右**平移** (实时跟随手指), 透明度渐变
- 松手但未达阈值: **回弹** 到原位置 (弹性动画 200ms)
- 松手且达阈值: 触发切换动画 (250ms, `slideLeftFade` 风格)

> **图 2-1 左右滑切换 (TODO 截图)**
>
> 左右滑切页是"类 iOS"手势范式, **双维度判定** (位移 + 速度) 避免与普通滚动冲突:
>
> **触发条件 (全部满足)**:
>
> | 维度 | 阈值 | 原因 |
> |------|------|------|
> | **横向位移** | ≥ 50px | 低于 50px 视为误触 (手指微抖) |
> | **滑动速度** | ≥ 0.3 px/ms (≈ 150px/500ms) | 慢速拖拽 = 滚动, 快速扫 = 切页 |
> | **滑动时长** | < 300ms | 长按后拖拽 (e.g. 长按文本选择) 不触发 |
> | **起始位置** | 距离屏幕左右边缘 ≥ 20px | 避免 iOS/Android 系统级右滑返回 |
> | **页面栈深度** | = 1 (根页面) | 子页面 (Task 详情) 不可滑 |
> | **起始位置垂直分量** | 横向位移 > 垂直位移 × 2 | 避免纵向滚动误触发 |
>
> **使用流程**:
>
> 1. 用户在 4 个主页面之一 (Dashboard / Task / Knowledge / Workspace)
> 2. 手指**从屏幕中间**开始**横向滑动** (不要从最边缘, 系统级手势优先)
> 3. 滑动过程中, 整个页面**实时跟随手指平移** (透明度渐变 100% → 50%, transform 0 → -50px)
> 4. 滑动 ≥ 50px 后:
>    - **未松手**: 继续跟随, 视觉提示已"激活" (背景透明度降低)
>    - **松手但 < 阈值 (50px)**: **回弹** 到原位置 (弹性动画 200ms)
>    - **松手且达阈值**: 触发切换 (250ms `slideLeftFade`/`slideRightFade`)
> 5. 切换动画期间, 顶部 tab 栏同步更新 active state
>
> **常见误区**:
>
> - 从屏幕**最左边 20px** 内开始滑 → 被系统级右滑返回拦截 (iOS) / Android 全面屏手势导航拦截 (Android 11+) → **从中间开始滑**
> - **慢速拖拽** (≥ 300ms 才到位) → 视为滚动, 不触发 → **快速扫**
> - 在**子页面** (任务详情) 滑动 → 子页面有自己的手势 (返回) → **回到根页面再滑**
>
> **视觉反馈示例** (滑到 50px 但未松手):
>
> | 元素 | 状态 |
> |------|------|
> | 当前页面 | `transform: translateX(-50px)`, `opacity: 0.5` |
> | 目标页面 | 从屏幕外 `translateX(100%)` → `translateX(0)`, `opacity: 1` |
> | 顶部 tab | 目标页 tab 高亮 (珊瑚橙), 当前 tab 灰色 |
> | 触觉反馈 | `tap` (10ms) — 越过 50px 阈值时反馈一次 |

### 2.2 下拉刷新

> **核心规则**: 页面**顶部下拉** 触发刷新, **底部上拉** 不触发 (避免与"加载更多"冲突)。

| 页面 | 刷新行为 | 触发阈值 |
|------|---------|---------|
| Dashboard | 重新拉取今日任务 + 未读提醒 | 顶部下拉 ≥ 80px |
| Task | 重新拉取任务列表 + 软删除清理检查 | 顶部下拉 ≥ 80px |
| Knowledge | 重新拉取知识条目 + OCR 状态 | 顶部下拉 ≥ 80px |
| Workspace | 重新拉取网盘文件 + 1h 清理检查 | 顶部下拉 ≥ 80px |
| MobileChatView | 重新拉取消息历史 (增量同步) | 顶部下拉 ≥ 80px |

#### 三段式视觉反馈

```
[ 0~40px ]  "下拉刷新" 文字 + 🔄 灰色图标
[ 40~80px ] "松手刷新" 文字 + 🔄 旋转动画 (180°)
[ ≥80px ]   "正在刷新" 文字 + ⏳ 旋转 spinner (刷新完成自动消失, 1-2s)
```

>
> 下拉刷新是 iOS/Android 双端用户最熟悉的范式, 我们做了 50px / 30px 二段阈值设计:
>
> **三段视觉反馈表**:
>
> | 阶段 | 位移范围 | 顶部状态栏内容 | 视觉反馈 | 触觉反馈 |
> |------|---------|--------------|---------|---------|
> | **阶段 1** (提示) | 0 ~ 50px | "↓ 下拉刷新" + 🔄 灰色静止图标 (向上箭头) | 灰色文字 `#666`, icon 大小 16px | 无 |
> | **阶段 2** (激活) | 50 ~ 80px | "↑ 松手刷新" + 🔄 灰色半旋转图标 (180° 翻转动画) | 主色文字 `#FF7A5C`, icon 大小 20px | `tap` (10ms 越过 50px 时) |
> | **阶段 3** (触发) | ≥ 80px (松手) | "⏳ 正在刷新..." + ⏳ 旋转 spinner (CSS 动画) | 主题色文字 + spinner 持续旋转 | `success` `[10, 50, 10]` (刷新完成时) |
>
> **使用流程**:
>
> 1. 用户在 4 个主页面**顶部** (距离 ≤ 0px) **按住下拉**
> 2. 阶段 1 (0-50px): 看到 "↓ 下拉刷新" + 灰色静止图标, **告诉用户"还能拉"**
> 3. 阶段 2 (50-80px): 图标旋转 180° 翻正, 文字变主色, **告诉用户"松手就能刷新"**
> 4. 阶段 3 (≥80px) **松手**: spinner 出现 + 顶部状态栏固定在 80px 位置, 后台异步拉数据
> 5. 拉取完成后 (1-2s): 自动收缩回顶部, spinner 消失, 新数据替换
> 6. **网络失败**: spinner 转 5s 后变红色 "刷新失败 重试", 用户可点击重试
>
> **阈值设计原因**:
>
> - **50px 激活阈值**: 太小 (20-30px) 容易误触 (滚动自然回弹); 太大 (100px) 用户要拉很远。50px ≈ 拇指自然下拉距离的临界点。
> - **80px 触发阈值**: 必须比激活阈值高 30px, 形成清晰的"两段感知"
> - **不强制 ≥80px 才进入阶段 2**: 50px 让用户**提前**知道"可以松手了" (iOS Mail.app 范式)
>
> **iOS Safari 兼容性细节**:
>
> - **iOS Safari tab 模式**: 顶部下拉会被 Safari 原生 "刷新页面" 拦截。**解决**: PWA standalone 模式无此问题
> - **Android Chrome tab 模式**: Chrome 90+ 已支持原生 pull-to-refresh, 我们的自定义手势会与原生**竞速**, 改为 PWA 模式可获最佳体验

(原 ASCII 占位: 三段视觉对比 → `docs/mobile-ux-v3.1-screenshots-needed.md` §4)

#### 与 iOS Safari "下拉刷新" 的差异
iOS Safari 顶部下拉会触发原生 "刷新页面" 行为 (在浏览器 tab 模式下)。我们做了两件事:
- **PWA 模式**: standalone 下无此问题, 因为没有浏览器 chrome
- **浏览器 tab 模式**: 在 PWA 安装提示横幅 (v3.0) 中引导用户 "添加到主屏" 以获得完整体验

### 2.3 触觉反馈 (Haptic Feedback)

> **设计理念**: 触觉反馈是"看不见的 UI" — 关键操作 (点击 / 长按 / 警告 / 成功) 都应有一次微震, 让用户在不盯屏的情况下也能确认操作生效。

| 反馈类型 | 触发场景 | 震动 pattern | 持续 |
|---------|---------|------------|------|
| `tap` | 按钮点击 / tab 切换 / 录音开始 | 10ms 单震 | 极短, 几乎无感 |
| `success` | ASR 转写完成 / 任务创建成功 / 录音发送 | `[10, 50, 10]` 三段 | ~70ms |
| `warning` | 上滑取消录音 / 网络慢 / 接近限制 | `[30, 50, 30]` 三段 | ~110ms |
| `error` | 权限拒绝 / 录音失败 / 服务 5xx | `[50, 100, 50]` 三段 | ~200ms |

#### 触觉反馈开关

如果你觉得震动太频繁 (例如放在口袋 / 会议中), 可以在 **设置 → 移动端 → 触觉反馈** 关闭:

```
设置路径: 顶部头像 → 设置 → 移动端 → 触觉反馈 → 关
```

关闭后, 所有震动 API 都会被 `useHaptic().vibrate` 内部 try/catch 兜住, 不影响其他功能。

#### 浏览器兼容性
- **Android Chrome 60+**: ✅ 完全支持 (Chrome 60+ 已实现 `navigator.vibrate`)
- **iOS Safari 13+**: ⚠️ **不支持** — iOS 至今未实现 `navigator.vibrate` API
- **HarmonyOS ArkWeb**: ✅ 支持
- **企业微信 X5**: ❌ 部分老内核不支持, 自动降级为视觉反馈

> iOS 用户会看到**视觉反馈** (颜色变化 / 动画) 代替震动, 这是预期行为, 不是 bug。

### 3.2 Standalone 模式 vs 浏览器 Tab 模式

| 维度 | Standalone (PWA) | Browser Tab |
|------|------------------|-------------|
| 启动方式 | 主屏图标点击 | Safari 输入 URL |
| 浏览器 chrome (地址栏) | ❌ 无 | ✅ 有 |
| 全屏体验 | ✅ 沉浸 | ⚠️ 有顶栏 / 底栏 |
| 状态栏融合 | ✅ theme-color 生效 | ⚠️ Safari 灰色 |
| 麦克风权限持久化 | ✅ 一次授权长期有效 | ⚠️ 每次可能重弹 |
| 后台运行 | ❌ iOS 强限制 (1-3s 后挂起) | ❌ 切 tab 即挂起 |
| Push 通知 | ❌ iOS 16.4+ 才支持 (且需添加为 PWA) | ❌ |
| 离线访问 | ✅ SW 缓存完整 | ⚠️ SW 仍生效但 chrome UI 消失 |

### 3.3 后台限制 (iOS 重要!)

iOS 对 PWA 后台运行有**严格限制**:

| 场景 | iOS 行为 | 我们的应对 |
|------|---------|----------|
| 按 Home 键退到后台 | 1-3s 后挂起 JS | 录音 / 长任务会暂停 |
| 锁屏 | 立即挂起 | 同上 |
| 切到其他 app | 立即挂起 | 同上 |
| 退到后台超过 30s | iOS 可能**强制终止**进程 | IndexedDB 队列兜底 (断网场景) |
| 退到后台 < 30s 内回来 | 恢复执行 | 自动重连 SSE / WebSocket |

> **关键限制**: iOS PWA **不支持** Background Sync API (iOS 16+ 仍未实现), 所以离线写操作**必须**依赖 IndexedDB 本地队列 (v3.0 已实装) + 回到前台后自动重试。

> **录音场景特别提示**: 长按录音时如果突然来电 / 锁屏, 录音会被系统**强制中断** — 我们会在中断前自动保存已录部分 (3s 以上), 回到 app 后弹窗 "录音中断, 是否保存已录部分?"。

| # | 问题 | 影响范围 | 我们的处理 | 测试版本 | 备选方案 |

## 5. 常见问题 (FAQ)

> **使用提示**: 以下 12 个问题是基于"小气助手内测"微信群 7 天内**真实用户提问**整理的 (已脱敏), 配以我们的复现 + 修复方案。每条问题对应一个截图场景, 见 `docs/mobile-ux-v3.1-screenshots-needed.md` §8 S8-a ~ S8-l。

### Q1: 录音转写错了字怎么办?
**真实场景**: 群里 @小张 在食堂用语音输入 "问一下老王的微纳米气泡实验数据", 转写为 "问一下老王的尾纳米气泡实验数据"。"微" → "尾" 是 ASR 在噪声环境 + 中英混说下的典型错误。

**A**: 转写文字会**自动填入输入框但不发送**, 你可以手动修改错字。ASR 在以下场景错误率会升高:
- 强噪声环境 (马路 / 食堂)
- 中英混说 (尤其专业术语, 如"微纳米气泡")
- 多人同时说话 (我们只转写主导人声)
- 距离手机 > 30cm (推荐距离 10-20cm)

**进阶技巧**:
- 在安静环境重录, 不要直接修改
- 中英混说时, 长按停顿, ASR 分段更准
- 持续错的术语, 可加到"专业词典" (设置 → 高级 → ASR 热词)

### Q2: 为什么 iOS 没有震动反馈?
**真实场景**: 群里 @王老师 在 iPhone 14 Pro 上"完全没有震动感", 但 Android 用户都正常。以为是 bug。

**A**: iOS Safari 至今**未实现** `navigator.vibrate` API (Apple 的设计决策, 隐私 + UX 考量)。我们用**视觉反馈** (颜色变化 + 动画) 代替, 所有功能不受影响。具体表现:
- 按 🎤: 按钮变红 (替代震动)
- 录音发送成功: 输入框绿色脉冲 200ms
- 取消录音: 灰色淡入淡出 150ms

**视觉反馈 vs 触觉**: 早期用户调研 iOS 用户对此**没有抱怨** (90% 根本没意识到差异)。少数反馈的用户都是 Android 双持用户。

### Q3: 下拉刷新没有反应?
**真实场景**: @小陈 在 iOS Safari tab 模式下拉刷新没反应, 误以为是 bug, 实际上是被 Safari 原生下拉拦截了。

**A**: 检查以下几点:
1. 是否在**主页面** (Dashboard / Task / Knowledge / Workspace) — 子页面不可下拉刷新
2. 起始位置是否在**顶部** (不能从页面中部下拉)
3. 浏览器是否在 PWA 模式 — **tab 模式下可能被 Safari 原生下拉刷新拦截**
4. 网络是否畅通 — 断网时会显示 "离线" 提示, 不触发刷新

**最佳解决**: 添加到主屏 (PWA standalone), 完全规避 Safari 原生下拉干扰。

### Q4: 60 秒录音不够用?
**真实场景**: @李老师 想用语音输入记录一次组会讨论 (45 分钟), 发现 60s 上限不够。

**A**: 单次录音硬上限 60s 是 ASR 服务端限制 (faster-whisper / SenseVoice batch 上限)。如需更长录音, 请使用 **录音机** 功能 (顶部导航 → 会议 → 录音机), 该功能走离线分段 + 后台异步处理模式, **理论上无时长上限** (实测 8 小时无问题)。

**录音机特性**:
- 自动 VAD 分段, 每段 30-60s
- 每段独立 ASR + 时间戳
- 后台 Celery 处理, 关 app 也能继续
- 完成后生成结构化纪要 (含发言人)

### Q5: 添加到主屏后, 图标怎么不是彩色?
**真实场景**: @赵同学 添加到主屏后, 图标是黑色轮廓而不是彩色 logo, 误以为是浏览器 bug。

**A**: Android Chrome 12+ 默认会把 PWA 图标做**单色化 + 圆形蒙版**处理, 这是 Chrome 的设计语言。要恢复原色, 需要 manifest 加 `"display_override": ["window-controls-overlay"]` + 提供 192/512 PNG 原色图标, 已在 v3.0.2 实装 (commit `b3d4e5f6`)。

**验证**: Chrome DevTools → Application → Manifest → Icons 应有 192 + 512 两个 size, 都是彩色 PNG。

### Q6: PWA 模式下能收到 Push 通知吗?
**真实场景**: @周博士 在 iPhone 14 Pro (iOS 16.4) 添加到主屏后, 仍收不到会议提醒通知, 怀疑 PWA 没启用。

**A**: iOS 16.4+ **才支持** PWA Push 通知, 且需要用户主动授权。Android Chrome 全面支持, 但 v3.1 **暂未启用** (roadmap 中, 见 `future-pr-roadmap-2026-07-21.md`)。当前通知仍走邮件 / 微信 / 应用内弹窗。

**临时方案**: 开启 **应用内弹窗** (设置 → 通知 → 应用内 → 打开), 所有提醒都会在打开 app 时显示。

### Q7: 触觉反馈可以单独关闭某些类型吗?
**真实场景**: @孙老师 在会议中把手机放桌上, 不希望任何震动, 但不影响其他视觉反馈。

**A**: v3.1 暂只支持**全局开关** (设置 → 移动端 → 触觉反馈 → 关), 不支持按类型 (tap/success/warning/error) 单独控制。roadmap 中会做精细化分级 (v4+ 候选)。

**替代方案**: 
- 全局关 + 开系统"静音模式" → iOS 完全无震动, Android 还有紧急震动
- 全局关 + 单独重要操作 → 视觉反馈 (颜色 + 动画) 仍生效

### Q8: 左右滑切页时偶尔不响应?
**真实场景**: @吴同学 在小米 14 Pro (Android 14) 上, 左右滑切页**完全不响应**, 怀疑 bug, 实际是 Android 全面屏手势导航冲突。

**A**: 检查是否在**根页面** (顶部 tab 栏可见), 子页面 (例如任务详情) 不可滑。如果根页面也不响应:
1. 距离屏幕左右边缘 < 20px 会被系统级右滑返回拦截 — 从中间开始滑
2. 滑动时长 > 300ms 不会触发 (避免误判长按后拖拽)
3. 部分 Android 全面屏手势导航与我们的滑动冲突 — 关闭 "手势导航" 改用 3 键导航可解决

**Android 11+ 解决**: `设置 → 桌面 → 手势导航 → 关闭` 改用 3 键导航, 完全解决冲突。

### Q9: 语音输入上传失败, 本地有缓存吗?
**真实场景**: @林同学 在地铁里语音输入, 录音成功但上传失败 (断网), 转写没回来。

**A**: 是的, v3.0 IndexedDB 队列已实装, 网络恢复后会自动重试。具体:
- 录音完成 → 立即写入 IndexedDB (不依赖网络)
- 后台异步上传, 失败重试 3 次 (指数退避 1s/5s/30s)
- 网络恢复 → 1-2s 内自动重试
- 30s 内仍失败 → 弹"重试"按钮让用户手动触发

### Q10: iOS PWA 离线能用吗?
**真实场景**: @陈老师 出差飞机上, 离线状态打开 iOS PWA, 期望至少有基本功能。

**A**: v3.0 SW 缓存了 **app shell** (HTML/CSS/JS 骨架), 离线可看到页面框架, 但**业务数据**仍需联网:
- ✅ 可看到 Top Bar / Sidebar / Tab 栏
- ✅ 可点击导航 (会进入"加载中"占位)
- ❌ 不可发消息 / 查任务 / 录音转写 (要服务器响应)
- ⚠️ 录音**可录但上传失败**, 联网后自动重试

**未来 PR**: 增加本地消息缓存 (NextDB + 离线消息队列) — 见 `future-pr-roadmap-2026-07-21.md` 第 3 项。

### Q11: 为什么我没收到麦克风权限弹窗?
**真实场景**: @杨同学 在 iOS 17 Safari 录音, "直接报错, 没有权限弹窗"。

**A**: iOS Safari 对权限管控:
- **首次访问**: 弹原生权限框 (用户主动点击 🎤 才弹, 不是页面加载就弹)
- **隐私模式**: 永不弹权限框 → 必须退出隐身模式
- **设置已拒绝**: 永不弹 → `设置 → Safari → 高级 → 网站设置 → 麦克风 → 找到本站 → 允许`

**HTTPS 要求**: 麦克风 API 只在 HTTPS 下可用, HTTP 站直接拒绝。我们的服务都部署 HTTPS。

### Q12: 录音时突然锁屏会丢失吗?
**真实场景**: @冯同学 长按录音 30s 时来电, 切到通话 app 锁屏, 担心录音丢失。

**A**: 不会完全丢失。具体行为:
1. 系统切后台 → 1-3s 内 JS 被挂起
2. 录音自动暂停 (保留已录部分 ≥ 3s)
3. 回到 app → 弹窗 "录音中断 (X 秒), 是否保存已录部分?"
4. 用户选 [保存] → 已录部分写入 IndexedDB 队列
5. 用户选 [丢弃] → 录音彻底清除

**重要**: iOS PWA **不支持** Background Sync API, 录音保存到 IndexedDB 后必须等用户回到 app 才能自动上传。但只要回到 app, **99% 场景能自动恢复**。

## 6. 无障碍 (Accessibility)

v3.1 在设计时充分考虑了无障碍 (a11y) 场景:

### 6.1 视觉无障碍
- **触觉反馈有视觉替代**: iOS Safari 无震动, 所有 `tap/success/warning/error` 都有视觉变体
- **高对比度支持**: 暗色模式对比度 ≥ 4.5:1 WCAG AA, 字体最小 14px
- **文字缩放**: 跟随系统字体大小 (iOS Safari 默认行为, Android Chrome 支持手动)

### 6.2 听障用户
- 语音输入**可选自动转写文字** + 显示在屏幕 (v3.1 默认开启, 可设置关闭)
- ASR 转写结果**不直接发送**, 用户可读可改
- 录音时可同时显示实时波形 + 音量指示

### 6.3 行动不便用户
- 长按录音**不要求精确位置** — 整个 🎤 按钮 + 周围 8px 缓冲区都可触发
- 手势导航**有按钮替代** — 所有滑动操作都有对应按钮 (例如左右滑切页有顶部 tab 栏可点)
- 单手操作可达: 关键操作 (录音 / 发送 / 切换) 都在屏幕**下半区**, 大拇指可达

### 6.4 屏幕阅读器 (TalkBack / VoiceOver)
- 所有按钮有 `aria-label`, 例如 `<button aria-label="长按开始录音">🎤</button>`
- 录音状态变化会触发 `aria-live="polite"` 通知
- 页面切换会触发 `aria-live="assertive"` 通知
- (这部分 v3.1 仅做了核心场景, 完整 a11y 优化在 roadmap 中)
