## 1. 语音输入 (G-1)

语音输入是 v3.1 的核心移动端体验升级 — 在 `MobileChatView` / `MobileCommandPalette` / `MobileSearchSheet` 三个高频输入场景默认开启, 让"边走边说"成为可能。

### 1.1 三态交互 (按住 / 上滑 / 松开)

| 状态 | 触发动作 | 视觉反馈 | 触觉反馈 |
|------|---------|---------|---------|
| **空闲** (idle) | 看到 🎤 按钮 (在输入框右侧) | 灰色实心话筒 | 无 |
| **录音中** (recording) | **长按** 🎤 0.3s 触发 | 红色脉冲圈 + 实时波形 + 计时器 | `tap` (10ms 短震) |
| **取消中** (cancelling) | 按住时**向上滑**超过 80px | "松手取消"气泡 + 录音图标淡灰 | `warning` `[30, 50, 30]` |
| **发送** (sending) | 松开手指 (未上滑取消) | 波形收尾 + 自动转文字 | `success` `[10, 50, 10]` |

### 1.2 详细操作步骤

#### 步骤 1: 长按开始录音
1. 打开小气助手 → 进入 `MobileChatView` (或任何带 🎤 的页面)
2. 找到输入框右侧的 🎤 麦克风图标
3. **长按** 约 0.3 秒 — 会感受到一次轻微震动 (`tap`)
4. 松开长按前的"等待"期: 系统正在请求麦克风权限 (首次会弹原生权限框)
5. 权限授予后: 录音正式开始, 图标变红 + 实时波形显示

#### 步骤 2 (可选): 上滑取消
- 如果说错话了, 不要松开, **继续按住并向上滑动** 超过 80px
- 输入框上方出现 "松手取消" 气泡
- 此时松开手指, 录音被丢弃 (不发送, 不入 ASR 队列)
- **常见误区**: 有些人会习惯性下滑试图"撤回", 这在我们的设计里**没有效果** — 只有上滑才算取消

#### 步骤 3: 松开发送
- 正常说完后, **直接松开手指** (不要上滑)
- 系统自动停止录音 + 上传到 ASR 服务
- ASR 转写完成后, 文字自动填入输入框 (不直接发送, 你可以再编辑/补字)
- 文字出现时伴随一次 `success` 触觉反馈, 确认转写成功

#### 步骤 4 (可选): 二次编辑
- 转写文字会**自动填入**输入框但**不发送**, 你可以:
  - 直接按 ➤ 发送
  - 手动修改错字 (ASR 在中英混说 / 专业术语 / 噪声环境下会有 5-10% 错误率)
  - 继续按 🎤 追加录音 (最长单次 60s, 超过会自动分段)

### 1.3 iOS Safari 权限流程

iOS Safari 对麦克风权限管控最严格, 首次录音会经历 3 步:

| 步骤 | 触发 | UI 表现 | 用户操作 |
|------|------|---------|---------|
| 1 | 长按 🎤 | 浏览器原生权限弹窗: "允许 'xxx.xx' 使用麦克风?" | 点 "允许" / "不允许" |
| 2 | 允许后 | 状态栏顶部出现 🎙️ 红色图标 (系统级, 不可关闭) | 无需操作, 表示正在录音 |
| 3 | 首次以后 | 直接进入录音中状态 (无弹窗) | 无需操作 |

> **⚠️ iOS Safari 14+ 特别说明**:
> - 在 **非 PWA (浏览器 tab)** 模式下, 每次进入页面都可能需要重新授权 (iOS 14.5+ 隐私策略)
> - 在 **PWA standalone 模式** 下, 授权会**持久化** — 一次授权, 长期有效 (添加到主屏后)
> - 如果误点了 "不允许", 需要去 `设置 → Safari → 高级 → 网站设置 → 麦克风 → 找到本站 → 允许`
> - **HTTPS 强约束**: 麦克风 API 只在 HTTPS 下可用, HTTP 站会直接拒绝 — 我们的服务都部署在 HTTPS + Nginx 反代后

### 1.4 录音时长限制

| 限制 | 数值 | 原因 |
|------|------|------|
| 单次最长 | 60 秒 | 移动端 ASR 服务端单请求 60s 硬上限 (faster-whisper / SenseVoice batch 上限) |
| 单次最短 | 0.5 秒 | 低于此时长视为误触, 直接丢弃 |
| 每日上限 | 无 | 走 Celery 队列, 不限频次 (但高峰时段排队可能 5-10s) |
| 文件大小 | ~500KB / 60s | Opus codec @ 32kbps, 移动端 4G 网络下 < 1s 上传完成 |

> 如果你需要录制超过 60 秒的长内容 (例如会议录音), 请使用 **录音机** 功能 (顶部导航 → 会议 → 录音机), 该功能是离线分段 + 后台异步处理模式, 不受 60s 限制。

### 1.5 ASR 转写引擎选择

v3.1 默认使用 **SenseVoice** (阿里达摩院开源, 2026-06-30 benchmark 5/5 胜 faster-whisper), 自动按以下优先级降级:

```
SenseVoice (主) → faster-whisper (备) → Paraformer (兜底) → 本地缓存错例
```

中文 / 中英混说 / 噪声环境表现最佳; 纯英文 / 其他语种自动降级到 faster-whisper。

### 1.6 错误处理与降级

| 错误场景 | UI 表现 | 降级方案 |
|---------|---------|---------|
| 麦克风权限被拒 | "请允许麦克风权限" 提示 + 设置引导链接 | 引导去系统设置 |
| 浏览器不支持 MediaRecorder | 🎤 按钮变灰 + tooltip "当前浏览器不支持" | 仅企业微信 X5 / HarmonyOS ArkWeb 等极老内核, 普通用户遇不到 |
| getUserMedia 5s 超时 | "无法访问麦克风, 请检查系统权限" | HarmonyOS / 企业微信已知问题, 引导刷新或换浏览器 |
| ASR 服务 5xx | "语音转文字失败, 请重试或手动输入" | 自动重试 1 次 + 保留本地音频, 可手动触发重试 |
| 网络断网 | 录音正常完成, 排队到 IndexedDB, 恢复后自动重试 | v3.0 IndexedDB 队列兜底 |

## 2. 手势导航 (G-2)

v3.1 在 v3.0 长按菜单 + 响应式断点的基础上, 进一步引入 iOS 风格的"滑切"手势, 让单手操作成为可能。

### 2.1 左右滑切换页面

> **核心规则**: 在 **Tab 栏同级** 的 4 个主页面 (Dashboard / Task / Knowledge / Workspace) 之间左右滑切换, 不影响子页面栈。

| 起始页 | 左滑 → | 右滑 → | 备注 |
|--------|--------|--------|------|
| Dashboard | Task | (无) | 第 1 页, 右滑无效 |
| Task | Knowledge | Dashboard |  |
| Knowledge | Workspace | Task |  |
| Workspace | (无) | Knowledge | 末页, 左滑无效 |

#### 触发条件
- **横向位移** ≥ 50px
- **滑动时长** < 300ms (避免长按后拖拽被误判)
- **起始位置** 距离屏幕左右边缘 ≥ 20px (避免误触系统级右滑返回)
- **页面栈深度 = 1** (根页面), 子页面 (例如点入 Task 详情) 不可滑

#### 视觉反馈
- 滑到 ≥ 50px 但未松手: 整个页面向左/右**平移** (实时跟随手指), 透明度渐变
- 松手但未达阈值: **回弹** 到原位置 (弹性动画 200ms)
- 松手且达阈值: 触发切换动画 (250ms, `slideLeftFade` 风格)

### 2.2 下拉刷新

> **核心规则**: 页面**顶部下拉** 触发刷新, **底部上拉** 不触发 (避免与"加载更多"冲突)。

| 页面 | 刷新行为 | 触发阈值 |
|------|---------|---------|
| Dashboard | 重新拉取今日任务 + 未读提醒 | 顶部下拉 ≥ 80px |
| Task | 重新拉取任务列表 + 软删除清理检查 | 顶部下拉 ≥ 80px |
| Knowledge | 重新拉取知识条目 + OCR 状态 | 顶部下拉 ≥ 80px |
| Workspace | 重新拉取网盘文件 + 1h 清理检查 | 顶部下拉 ≥ 80px |
| MobileChatView | 重新拉取消息历史 (增量同步) | 顶部下拉 ≥ 80px |

#### 三段式视觉反馈

```
[ 0~40px ]  "下拉刷新" 文字 + 🔄 灰色图标
[ 40~80px ] "松手刷新" 文字 + 🔄 旋转动画 (180°)
[ ≥80px ]   "正在刷新" 文字 + ⏳ 旋转 spinner (刷新完成自动消失, 1-2s)
```

#### 与 iOS Safari "下拉刷新" 的差异
iOS Safari 顶部下拉会触发原生 "刷新页面" 行为 (在浏览器 tab 模式下)。我们做了两件事:
- **PWA 模式**: standalone 下无此问题, 因为没有浏览器 chrome
- **浏览器 tab 模式**: 在 PWA 安装提示横幅 (v3.0) 中引导用户 "添加到主屏" 以获得完整体验

### 2.3 触觉反馈 (Haptic Feedback)

> **设计理念**: 触觉反馈是"看不见的 UI" — 关键操作 (点击 / 长按 / 警告 / 成功) 都应有一次微震, 让用户在不盯屏的情况下也能确认操作生效。

| 反馈类型 | 触发场景 | 震动 pattern | 持续 |
|---------|---------|------------|------|
| `tap` | 按钮点击 / tab 切换 / 录音开始 | 10ms 单震 | 极短, 几乎无感 |
| `success` | ASR 转写完成 / 任务创建成功 / 录音发送 | `[10, 50, 10]` 三段 | ~70ms |
| `warning` | 上滑取消录音 / 网络慢 / 接近限制 | `[30, 50, 30]` 三段 | ~110ms |
| `error` | 权限拒绝 / 录音失败 / 服务 5xx | `[50, 100, 50]` 三段 | ~200ms |

#### 触觉反馈开关

如果你觉得震动太频繁 (例如放在口袋 / 会议中), 可以在 **设置 → 移动端 → 触觉反馈** 关闭:

```
设置路径: 顶部头像 → 设置 → 移动端 → 触觉反馈 → 关
```

关闭后, 所有震动 API 都会被 `useHaptic().vibrate` 内部 try/catch 兜住, 不影响其他功能。

#### 浏览器兼容性
- **Android Chrome 60+**: ✅ 完全支持 (Chrome 60+ 已实现 `navigator.vibrate`)
- **iOS Safari 13+**: ⚠️ **不支持** — iOS 至今未实现 `navigator.vibrate` API
- **HarmonyOS ArkWeb**: ✅ 支持
- **企业微信 X5**: ❌ 部分老内核不支持, 自动降级为视觉反馈

> iOS 用户会看到**视觉反馈** (颜色变化 / 动画) 代替震动, 这是预期行为, 不是 bug。

## 3. iOS Safari PWA 适配

iOS Safari 对 PWA 的支持**最严苛也最碎片化** — v3.0 已经做了 4 策略 (manifest + safe-area + service worker + useSafeArea), v3.1 在此基础上补充了 3 个细节。

### 3.1 添加到主屏

#### 推荐流程 (3 步)

1. **打开小气助手网站** (iOS Safari)
2. 点击底部 **分享按钮** (📤 方框带向上箭头)
3. 滑动找到 **"添加到主屏幕"** (Add to Home Screen), 点击
4. 确认名称 (默认 "小气助手"), 点击右上角 **"添加"**
5. 主屏出现 "小气助手" 图标, 点击即以 **standalone 模式** 启动

### 3.2 Standalone 模式 vs 浏览器 Tab 模式

| 维度 | Standalone (PWA) | Browser Tab |
|------|------------------|-------------|
| 启动方式 | 主屏图标点击 | Safari 输入 URL |
| 浏览器 chrome (地址栏) | ❌ 无 | ✅ 有 |
| 全屏体验 | ✅ 沉浸 | ⚠️ 有顶栏 / 底栏 |
| 状态栏融合 | ✅ theme-color 生效 | ⚠️ Safari 灰色 |
| 麦克风权限持久化 | ✅ 一次授权长期有效 | ⚠️ 每次可能重弹 |
| 后台运行 | ❌ iOS 强限制 (1-3s 后挂起) | ❌ 切 tab 即挂起 |
| Push 通知 | ❌ iOS 16.4+ 才支持 (且需添加为 PWA) | ❌ |
| 离线访问 | ✅ SW 缓存完整 | ⚠️ SW 仍生效但 chrome UI 消失 |
| safe-area-inset | ✅ 完整生效 | ✅ 也生效 |

> **结论**: 为了完整体验, **强烈建议添加到主屏** — 尤其需要使用语音输入 (G-1) 的用户。

### 3.3 后台限制 (iOS 重要!)

iOS 对 PWA 后台运行有**严格限制**:

| 场景 | iOS 行为 | 我们的应对 |
|------|---------|----------|
| 按 Home 键退到后台 | 1-3s 后挂起 JS | 录音 / 长任务会暂停 |
| 锁屏 | 立即挂起 | 同上 |
| 切到其他 app | 立即挂起 | 同上 |
| 退到后台超过 30s | iOS 可能**强制终止**进程 | IndexedDB 队列兜底 (断网场景) |
| 退到后台 < 30s 内回来 | 恢复执行 | 自动重连 SSE / WebSocket |

> **关键限制**: iOS PWA **不支持** Background Sync API (iOS 16+ 仍未实现), 所以离线写操作**必须**依赖 IndexedDB 本地队列 (v3.0 已实装) + 回到前台后自动重试。

> **录音场景特别提示**: 长按录音时如果突然来电 / 锁屏, 录音会被系统**强制中断** — 我们会在中断前自动保存已录部分 (3s 以上), 回到 app 后弹窗 "录音中断, 是否保存已录部分?"。

### 3.4 iOS Safari 已知兼容性问题

| 问题 | 触发版本 | 我们的处理 |
|------|---------|----------|
| `env(safe-area-inset-*)` 不生效 | iOS < 11 | fallback 到固定 20px padding |
| `100dvh` 单位不识别 | iOS < 15.4 | polyfill `100vh` → JS 读 `innerHeight` 二次校正 |
| `backdrop-filter` 性能差 | iOS 14-15 | 降级为半透明 background-color |
| `navigator.vibrate` 不支持 | 全版本 | useHaptic 内部 try/catch 静默 |
| `audio/webm` 不支持 | 全版本 | MediaRecorder MIME 探测链: webm;opus → ogg;opus → **mp4** (iOS 必须) |
| 麦克风权限 14.5+ 每次重弹 | iOS 14.5+ standalone 除外 | 引导用户添加到主屏 |

## 4. Android Chrome 适配

Android Chrome 是 PWA 体验**最佳**的平台 — v3.0 + v3.1 的所有功能都原生支持。

### 4.1 PWA 安装

Android Chrome 会在满足以下条件时**主动弹出** "添加到主屏" 提示:

1. manifest.webmanifest 完整 + MIME `application/manifest+json`
2. service worker 已注册 (含 fetch handler)
3. HTTPS 证书有效
4. 用户至少访问过 2 次, 间隔 ≥ 5 分钟
5. 浏览器检测到 "可安装" 信号 (Chrome 76+)

> **图 4-1 Android Chrome PWA 安装弹窗 (TODO 截图)**
>
> ```text

> └────────────────────────────────┘
> ```

#### 手动安装路径
如果弹窗没出现, 可以手动安装:
1. 浏览器右上角 **⋮** (三点菜单)
2. **"添加到主屏幕"** 或 **"安装应用"**
3. 确认名称 → 添加

### 4.2 通知权限

v3.1 暂未启用 Push 通知 (roadmap 中, 见 `future-pr-roadmap-2026-07-21.md`), 但**前端 Notification API** 已就绪:

```js
// 浏览器原生弹窗 (用户主动触发, 例如点了 "订阅会议提醒")
Notification.requestPermission().then(permission => {
  if (permission === 'granted') {
    new Notification('小气助手', { body: '会议 5 分钟后开始' })
  }
})
```

| 权限状态 | 表现 |
|---------|------|
| `default` (未询问) | 弹原生权限框: "允许 xxx.xx 发送通知?" |
| `granted` | ✅ 通知正常显示 |
| `denied` | ❌ 浏览器记住拒绝, 不会再次弹; 用户需去 `设置 → 网站 → 通知 → 找到本站 → 允许` |

### 4.3 后台音频

Android Chrome 对后台音频**最友好** — v3.1 的 PWA 音频播放 (TTS / 会议录音回放) 在后台仍能继续。

| 场景 | Android Chrome 行为 | iOS Safari 行为 |
|------|---------------------|------------------|
| TTS 播放时切后台 | ✅ 继续播放 | ⚠️ 立即暂停 |
| 会议录音回放时锁屏 | ✅ 锁屏控制 (上一首/下一首/暂停) | ⚠️ 立即暂停 |
| 后台音频焦点 | ✅ 与其他 app 音频正确混音/暂停 | ⚠️ 不一致 |

> **设计取舍**: 我们没有启用 Android Chrome 的 `MediaSession` 锁屏控制 (需要 `background-sync` 权限), 因为会议录音回放是**回看**行为, 用户通常不会在锁屏时操作。

### 4.4 Android Chrome 其他优势

| 能力 | 支持情况 |
|------|---------|
| `navigator.vibrate` | ✅ 完全支持 (Android Chrome 60+) |
| `MediaRecorder` (webm;opus) | ✅ 完全支持 (Android 5+) |
| Background Sync API | ✅ Chrome 49+ (但 PWA 不一定启用) |
| 离线缓存 (Cache API) | ✅ 完全支持 |
| Web Share API | ✅ 移动端 Chrome 61+ (v3.1 任务分享按钮可调起) |
| 触觉反馈 | ✅ 完整 (含 pattern) |

## 5. 常见问题 (FAQ)

### Q1: 录音转写错了字怎么办?
**A**: 转写文字会**自动填入输入框但不发送**, 你可以手动修改错字。ASR 在以下场景错误率会升高:
- 强噪声环境 (马路 / 食堂)
- 中英混说 (尤其专业术语)
- 多人同时说话
- 距离手机 > 30cm

### Q2: 为什么 iOS 没有震动反馈?
**A**: iOS Safari 至今未实现 `navigator.vibrate` API (Apple 的设计决策, 隐私 + UX 考量)。我们用**视觉反馈** (颜色变化 + 动画) 代替, 所有功能不受影响。

### Q3: 下拉刷新没有反应?
**A**: 检查以下几点:
1. 是否在**主页面** (Dashboard / Task / Knowledge / Workspace) — 子页面不可下拉刷新
2. 起始位置是否在**顶部** (不能从页面中部下拉)
3. 浏览器是否在 PWA 模式 — tab 模式下可能被 Safari 原生下拉刷新拦截
4. 网络是否畅通 — 断网时会显示 "离线" 提示, 不触发刷新

### Q4: 60 秒录音不够用?
**A**: 单次录音硬上限 60s 是 ASR 服务端限制 (faster-whisper / SenseVoice batch 上限)。如需更长录音, 请使用 **录音机** 功能 (顶部导航 → 会议 → 录音机), 该功能走离线分段 + 后台异步处理模式, **理论上无时长上限** (实测 8 小时无问题)。

### Q5: 添加到主屏后, 图标怎么不是彩色?
**A**: Android Chrome 12+ 默认会把 PWA 图标做**单色化 + 圆形蒙版**处理, 这是 Chrome 的设计语言。要恢复原色, 需要 manifest 加 `"display_override": ["window-controls-overlay"]` + 提供 192/512 PNG 原色图标, 已在 v3.0.2 实装。

### Q6: PWA 模式下能收到 Push 通知吗?
**A**: iOS 16.4+ 才支持 PWA Push 通知, 且需要用户主动授权。Android Chrome 全面支持, 但 v3.1 **暂未启用** (roadmap 中, 见 `future-pr-roadmap-2026-07-21.md`)。当前通知仍走邮件 / 微信 / 应用内弹窗。

### Q7: 触觉反馈可以单独关闭某些类型吗?
**A**: v3.1 暂只支持**全局开关** (设置 → 移动端 → 触觉反馈 → 关), 不支持按类型 (tap/success/warning/error) 单独控制。roadmap 中会做精细化分级。

### Q8: 左右滑切页时偶尔不响应?
**A**: 检查是否在**根页面** (顶部 tab 栏可见), 子页面 (例如任务详情) 不可滑。如果根页面也不响应:
1. 距离屏幕左右边缘 < 20px 会被系统级右滑返回拦截 — 从中间开始滑
2. 滑动时长 > 300ms 不会触发 (避免误判长按后拖拽)
3. 部分 Android 全面屏手势导航与我们的滑动冲突 — 关闭 "手势导航" 改用 3 键导航可解决

## 6. 无障碍 (Accessibility)

v3.1 在设计时充分考虑了无障碍 (a11y) 场景:

### 6.1 视觉无障碍
- **触觉反馈有视觉替代**: iOS Safari 无震动, 所有 `tap/success/warning/error` 都有视觉变体
- **高对比度支持**: 暗色模式对比度 ≥ 4.5:1 WCAG AA, 字体最小 14px
- **文字缩放**: 跟随系统字体大小 (iOS Safari 默认行为, Android Chrome 支持手动)

### 6.2 听障用户
- 语音输入**可选自动转写文字** + 显示在屏幕 (v3.1 默认开启, 可设置关闭)
- ASR 转写结果**不直接发送**, 用户可读可改
- 录音时可同时显示实时波形 + 音量指示

### 6.3 行动不便用户
- 长按录音**不要求精确位置** — 整个 🎤 按钮 + 周围 8px 缓冲区都可触发
- 手势导航**有按钮替代** — 所有滑动操作都有对应按钮 (例如左右滑切页有顶部 tab 栏可点)
- 单手操作可达: 关键操作 (录音 / 发送 / 切换) 都在屏幕**下半区**, 大拇指可达

### 6.4 屏幕阅读器 (TalkBack / VoiceOver)
- 所有按钮有 `aria-label`, 例如 `<button aria-label="长按开始录音">🎤</button>`
- 录音状态变化会触发 `aria-live="polite"` 通知
- 页面切换会触发 `aria-live="assertive"` 通知
- (这部分 v3.1 仅做了核心场景, 完整 a11y 优化在 roadmap 中)
