# Drive v2 PR9 用户指南 — 文件评论 & 版本历史 (2026-07-24)

> 面向课题组全体成员的使用教程。PR9 给网盘带来两个新能力:
> 1. **评论 thread** — 在文件/文件夹下讨论, 像 GitHub PR review / 腾讯文档评论一样, 支持嵌套回复、@ 提醒、"已解决"标记
> 2. **文件版本历史** — 每次上传新版本自动留档, 像 Google Drive / 坚果云一样, 随时下载旧版或一键回滚
>
> 移动端 (手机浏览器 / PWA) 与桌面端操作对照见各节表格。

---

## 1. 评论 thread 使用教程

### 1.1 我能在哪里评论?

- **文件评论**: 网盘中任意 drive 文件 (`storage_mode=drive`) 都可评论 — 论文 PDF、实验数据 Excel、组会 PPT 等
- **文件夹评论**: 文件夹级别整体讨论 (如 "本月论文归类怎么整理")
- **权限**: 你能**看到**这个文件/文件夹 (read 权限), 就能评论 — 类似 GitHub issue, 不需要编辑权限

### 1.2 发表评论 (移动端 vs 桌面端)

| 步骤 | 📱 移动端 | 🖥 桌面端 |
|------|-----------|-----------|
| 1. 进入评论 | 网盘 → 点开文件 → 点「评论」入口 (进入全屏评论页) | 网盘 → 文件行 → 评论图标 (侧栏/面板展开) |
| 2. 输入 | 底部固定输入框, 支持多行 | 面板底部输入框 |
| 3. @ 提醒 | 输入 `@` 弹出成员选择 | 输入 `@` 自动补全成员 |
| 4. 发送 | 点「发送」, 评论即时出现在列表顶部 | 回车或点发送 |

**@ 提醒**: 被 @ 的成员会在通知铃铛看到「XXX 在《文件名》评论中提到了你」, 点击直达。

### 1.3 回复与嵌套

- 点任意评论的「回复」→ 输入内容 → 发送, 回复挂在原评论下方缩进显示
- **嵌套不限层数** (回复的回复的回复…都可以), 与老版文件评论 (最多 3 层) 不同
- 📱 移动端: **长按评论**弹出操作菜单 (回复 / 编辑 / 删除), 带轻微振动反馈
- 🖥 桌面端: 悬停评论显示操作按钮

### 1.4 编辑与删除

| 操作 | 谁可以做 | 注意 |
|------|----------|------|
| 编辑自己的评论 | 仅评论作者本人 | 管理员也**不能**改别人的评论 (GitHub review 风格) |
| 删除自己的评论 | 仅评论作者本人 | ⚠️ 删除顶层评论会**连带删除其下所有回复**, 不可恢复 |

### 1.5 "已解决" 标记 (resolve)

讨论有结论后, 把 thread 标记为「已解决」, 列表默认可过滤掉已解决项, 保持界面干净。

- **谁能 resolve**: 评论作者 / 文件所有者 / 文件夹所有者 / 文件夹管理员 (任一即可)
- 已解决的 thread 显示 ✅ 标记 + 解决人; 点「重新打开」(unresolve) 可恢复讨论
- 筛选: 评论列表支持「全部 / 未解决 / 已解决」切换

### 1.6 典型场景示例

> **场景**: 师兄上传了《气泡粒径分布 v3.xlsx》到「实验数据」共享文件夹。
> 1. 你发现第 2 列数据异常 → 文件评论: 「第 2 列 3 月 15 日的数据是不是漏了单位换算? @王师兄」
> 2. 王师兄收到通知 → 回复: 「我查一下原始记录」→ 次日回复: 「确实, 已修正, 上传了新版本」(见第 2 节版本历史)
> 3. 你确认无误 → 点「已解决」→ thread 归档, 但记录永久保留可追溯

## 2. 文件版本历史教程

### 2.1 什么是版本历史?

同一个文件多次修改上传, 系统自动保留**每一版**的完整副本 (版本号 1, 2, 3… 递增)。主列表永远显示最新版, 历史版本收在「版本历史」面板, 不占列表位置。

### 2.2 上传新版本

| 步骤 | 📱 移动端 | 🖥 桌面端 |
|------|-----------|-----------|
| 1 | 文件详情 → 「版本历史」 | 文件行 → 更多菜单 → 「版本历史」 |
| 2 | 点「上传新版本」→ 选文件 | 点「上传新版本」→ 选文件或拖拽 |
| 3 | 可填版本备注 (如 "修正第 2 列单位") | 同左, 备注最长 500 字 |
| 4 | 确认 → 新版本成为当前版, 旧版进历史 | 同左 |

> 💡 **与直接重新上传的区别**: 在文件列表里重新上传会创建**新文件条目**; 走「上传新版本」则保持同一文件 (同一 ID、同一评论 thread、同一分享链接), 只更新内容。**推荐迭代中的文档一律走版本历史。**

### 2.3 查看与下载历史版本

- 「版本历史」面板按时间倒序列出所有版本: 版本号、上传人、时间、大小、备注、「当前」标记
- 点任意历史版本 → 「下载」→ 拿到该版完整文件 (不影响当前版)

### 2.4 回滚到旧版本

1. 版本历史 → 找到目标旧版 (如 v2) → 点「回滚到此版本」
2. 确认后系统**复制 v2 内容创建一个新版本** (如当前是 v5, 回滚后生成 v6, 内容 = v2)
3. 中间的 v3/v4/v5 **全部保留**, 不会丢 — 回滚永远可以再"回滚回来"

> ⚠️ 回滚**不是删除**: 版本号只增不减, 历史链完整可审计。

### 2.5 版本对比

- 当前版本: 直接对照两版的大小 / 上传人 / 备注 / 时间做人工判断; 需要内容级 diff 时, 分别下载两版本地对比
- 内容级自动 diff (文本 diff / 二进制 hash 对比) 排期在 PR10, 敬请期待

### 2.6 删除版本的限制

- 只能删除**头尾**版本, **禁止删除中间版本** (防止回滚链悬空)
- 「当前版本」不可删除 (先回滚/上传新版, 再删)

## 3. 常见问题 FAQ

### Q1: 我评论后对方没收到提醒?

评论本身**不会**给所有人发通知, 只有被 `@` 的成员才收到铃铛提醒。想让特定人看到, 记得输入 `@` 选择对方。文件所有者暂不自动订阅自己文件的所有评论 (WebSocket 实时推送排期 PR10)。

### Q2: 为什么我不能编辑/删除某条评论?

只有**评论作者本人**能编辑/删除自己的评论 — 管理员也不能替别人改 (设计如此, 对齐 GitHub review, 保证讨论记录可信)。如确有违规内容需要处理, 联系管理员从数据库层面处理。

### Q3: 上传新版本后, 之前的分享链接 / 评论会失效吗?

不会。版本历史的核心设计就是**文件身份不变**: 文件 ID、评论 thread、分享链接、所在文件夹全部保持, 只有文件内容切到新版。别人通过旧分享链接下载到的将是**当前版本** (即最新版); 需要给别人特定旧版, 从版本历史单独下载后发送。

### Q4: 历史版本占空间吗? 会被自动清理吗?

每个历史版本都是完整副本, 存在 MinIO 中, 会占用存储空间。当前**不会自动清理**历史版本 (删除文件时历史版本随之级联清除)。大文件 (视频/大数据集) 请避免频繁传新版本; 版本保留策略 (如只留最近 N 版) 排期在后续 PR。

## 4. 移动端专属提示

- 评论页是**全屏独立页面** (路由 `drive/file/{id}/comments`), 手机返回手势/返回键直接退回文件列表
- 长按操作带触觉振动反馈; 若无振动, 检查系统是否关闭了网页振动权限
- 深色模式: 评论页跟随 App 主题自动切换
- PWA 离线时评论**不可发送** (评论需实时到服务器), 恢复网络后重试即可
