## 8 条新铁律

### 铁律 1：tool_use / tool_result 必须严格配对（Anthropic API 平衡规则）
重检索 yield `tool_use` (id=`reretrieve_{n}_xxx`) 必须立即 yield `tool_result` (相同 id) + `messages.append` 两个 block (assistant tool_use + user tool_result)。否则下次 LLM 调用报 400。
**纪律**：复制 `agentic_loop.py:740-755` + `803-807` 错误兜底 pattern，**不**用 `try/except` 吞，必须 yield sentinel `tool_result: {status: error}` 保证 block 配对。

### 铁律 2：Self-RAG judge 必须 `default-on-fail` (parse-fail 返 can_answer=True, confidence=0.3)
LLM-as-judge 偶发非 JSON 响应 / 超时 / 5xx。**绝不让 judge 失败阻塞主流程**——返 fallback 走 `AGENT_SELF_RAG_RELAXED_THRESHOLD=0.4` 卡（fallback confidence 0.3 < 0.4 → 触发重检索/降级）。
**反向**：`can_answer=False, confidence=0.3` 这种组合**不要**默认返，否则会过于激进地重检索。

### 铁律 3：双层控制 per-request 优先于全局 settings
```python
self_rag_active = (
    ctx.self_rag_enabled if ctx.self_rag_enabled is not None
    else settings.AGENT_SELF_RAG_ENABLED
)
```
- 用户 toggle 关 → `ctx.self_rag_enabled = False` → gate 不跑
- 用户 toggle 开 → `ctx.self_rag_enabled = True` → gate 跑
- Ops 紧急关 → `.env` `AGENT_SELF_RAG_ENABLED=false` → 全局 kill switch

### 铁律 4：re-retrieve 必须按 rerank_score 降序合并去重
每次重检索的 `search_knowledge` 返回的 results 必须:
1. **去重** by `id`（已有 id 不重复 push）
2. **重排序** by `rerank_score` desc（高质 doc 排前）
3. **截断** to `AGENT_SELF_RAG_MAX_CONTEXT_DOCS=8`（避免 token 预算爆）
**否则**：低质 doc 挤掉高质 doc / token 撑爆 → LLM 答案退化。

### 铁律 5：refined_query ID 前缀 `reretrieve_{n}_xxx` 区别 Phase 1
- Phase 1 tool_use_id: `toolu_xxx` (Anthropic UUID)
- Phase 0.5 重检索 tool_use_id: `reretrieve_0_search_knowledge`, `reretrieve_1_search_knowledge`...
**目的**：前端可以靠 ID 前缀区分流式事件来源，关闭 `reretrieving` 动画时机精准。

### 铁律 6：gate 只对 search_info / explain_concept 触发
其他 intent（casual_chat / data_query / execute_action / recommend_person）**不**跑 Self-RAG gate。
- casual_chat → 无需检索
- data_query / execute_action → 走其他工具，无需 judge
- recommend_person → 推荐人不是检索评估范围
**目的**：节省 ~300ms judge latency + ~$0.001 cost 在非检索场景。

### 铁律 7：user_message 必须传到 `loop.run(user_message=...)`
Phase 0.5 gate 需要 `user_message` 做 judge prompt 输入。`messages` 是 list of dict 不能直接用。
**纪律**：`chat_engine.synthesize_stream` 内 `user_message=_last_user_text(messages)` 抽出，传入 `loop.run`。
**降级**：`user_message=""` 时不跑 gate（兜底）。

### 铁律 8：dark mode 跨组件覆盖必须非 scoped `<style>` 块
新增的 `.depth-toggle.active` / `.retrieval-badge` 必须放在 ChatViewSSE.vue 末尾**非 scoped** `<style>` 块的 `[data-theme="dark"]` 块中。
**踩坑历史**：v60-v67 教训 #N 次强化。Vue scoped 编译器把 `[data-theme="dark"]` + 后代选择器处理错，规则失效。
**纪律**：dark 块顶部加 `/* v60-v67 教训：必须非 scoped */` 注释一眼可见。

## 6 大踩坑教训

### 踩坑 1：tool_use 没配 tool_result → Anthropic API 报 400
重检索代码第一版用 `try/except: pass` 吞掉 dispatch 失败 → 不 yield tool_result → messages 不平衡 → 下次 LLM 调用 400。
**修复**：复制 `agentic_loop.py:740-755` 错误兜底 pattern，**必须** yield `tool_result: {status: error}` 保证 block 配对。
**测试覆盖**：`test_self_rag_message_balance` 必须 assert `len(tool_use_ids) == len(tool_result_ids)`。

### 踩坑 2：judge 延迟 +800ms 用户感知明显
Sonnet 4.6 第一次跑 judge 花了 850ms，每次 query 都加 800ms → 用户说"AI 变慢了"。
**修复**：默认用 Haiku 4.5 (~300ms) + `AGENT_SELF_RAG_MODEL` 配置项可切。
**调试技巧**：Claude dashboard 看 `Mean TTFT` 指标。

### 踩坑 3：refined_query 比 original 差
LLM 改写 v1 试过 → 改写后 query 太长 / 关键词不聚焦 → 检索质量反而降低。
**修复**：v1 用 deterministic 关键词拼接 + intent.keywords 优先 + 停用词剥除。LLM 改写留 #009c A/B。

### 踩坑 4：re-retrieve 合并新结果时 id 冲突
新旧 results 可能有相同 id（如同一篇 doc 跨多次检索命中）→ 直接 push 会重复。
**修复**：`existing_ids = {r.get("id") for r in search_k_results if r.get("id") is not None}` 去重。

### 踩坑 5：user_message 没传到 gate
`loop.run(messages, system, intent, ctx)` 拿不到纯文本 query → gate 拿到的 user_message 是空字符串 → judge 返 `can_answer=False, confidence=0.3` → 误触发重检索。
**修复**：`chat_engine.synthesize_stream` 调 `_last_user_text(messages)` 抽出来，传入 `loop.run(user_message=...)`。

### 踩坑 6：default-on-fail 让 bad result 通过
parse-fail 返 `can_answer=True, confidence=0.3` → 看起来还行 → 不重检索 → 用户拿 bad answer。
**修复**：`AGENT_SELF_RAG_RELAXED_THRESHOLD=0.4` 卡，0.3 < 0.4 触发重检索。
**纪律**：parse-fail confidence 必须**严格**低于 RELAXED_THRESHOLD。
