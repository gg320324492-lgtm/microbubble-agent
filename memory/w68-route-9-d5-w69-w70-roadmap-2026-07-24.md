## 6. 3 新铁律沉淀

### 铁律 1: 季度排期必须基于 plans 调研 + 拍板选项矩阵

**内容**: 季度排期 (W69 + W70) 必须有:
1. plans 调研依据 (W66 67 plans + W68 14 plans 留 W69 排期)
2. 派工选项矩阵 (选项 A / B / C, 派工数 + 时间 + 风险 + 用户价值 + 锚点范式预期)
3. 失败决策树 (单 agent / 整批失败回滚路径)
4. 主指挥拍板建议 (基于 4 倾向: 推进 / 跨主题 / 稳一手 / 资源受限)

**反模式**: 直接拍板派工数 + 时间, 无 plans 调研依据 + 无失败回滚路径 → 重现 W67 第 47 步 11 次循环教训.

**触发场景**: 季度排期决策 (W69 + W70 / W71 + W72 / ...).

### 铁律 2: 小修必须分散派工, 不允许单 agent 集中 5+ plans

**内容**: 小修 (plans Status 修正 + 调研发现 + 1 file 改动) 必须:
1. **分散**: 单 agent 处理 2-3 plans, 不超过 3 plans/agent (避免单 agent 工作量过大)
2. **并行**: 多个小修 agent 在同批次并行 (例如 W69 第 1 批 Agent #6 与 Agent #1 + #2 并行)
3. **scope 限定**: 单 agent 仅修改 plans Status 段 + 1 script 文件, 不实施 plan body (避免引出更大改动)

**反模式**: 单 agent 集中处理 5+ plans → 容易引出"实施 plan body"超出 scope → 0 production code 铁律破坏.

**触发场景**: W68 第 4-8 批小修实战 (Agent #6 + #7 + #8 三批小修分散派工验证).

### 铁律 3: 主指挥拍板选项 A/B/C 必须含 (派工数 + 时间 + 风险 + 用户价值 + 锚点范式预期) 5 维

**内容**: 主指挥拍板选项矩阵必须含 5 维:
1. **派工数**: 具体 agents 数量 (例如 5-7 agents)
2. **时间**: 预估完成时间 (例如 2 周)
3. **风险**: 风险等级 (低/中/中-高)
4. **用户价值**: 用户痛点解决程度 (例如子 plan ② 落地 + 6 plans 小修)
5. **锚点范式预期**: 锚点范式数值变化 (例如 W68 90 → W69 95-100 → W70 100-115)

**反模式**: 仅"派工数 + 时间"2 维 → 主指挥无法权衡风险与用户价值.

**触发场景**: 季度排期决策 + 单批派工决策 (W68 第 4 批 / 第 5 批 / W69 第 1 批 等).
