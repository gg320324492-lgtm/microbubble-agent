## 5 新铁律沉淀

### 铁律 1: 嵌套 path 重建必须先修孤儿

`parent_id` 指向不存在 ID 的评论必须在 `WITH RECURSIVE` 之前先剪枝 (置 NULL),
否则起点选不到这棵子树, 留 path=NULL 漂移。

```sql
-- 1. 先修孤儿 (置 parent_id NULL + path '/')
UPDATE drive_comments dc
SET parent_id = NULL, path = '/', depth = 0
WHERE dc.parent_id IS NOT NULL
  AND NOT EXISTS (SELECT 1 FROM drive_comments p WHERE p.id = dc.parent_id);

-- 2. 再跑 WITH RECURSIVE 重算
```

**事故**: W68 第 8 批 PR11 上线后, 5 个历史评论 path=NULL 漂移 7 天, 因为
066 migration 体内没修孤儿。W68 第 12 批 B-1 PR14 补这一步。

### 铁律 2: WITH RECURSIVE 必须有 depth 上限

`cp.depth < 100` 硬上限防循环引用导致无限递归。`UNION ALL` 不像 `UNION` 自动去重,
循环会爆 PostgreSQL 内存 (Err: out of memory / recursive query 太多层)。

```sql
WITH RECURSIVE comment_path_calc AS (
    SELECT ... WHERE parent_id IS NULL
    UNION ALL
    SELECT ... FROM drive_comments c
    INNER JOIN comment_path_calc cp ON c.parent_id = cp.id
    WHERE cp.depth < 100  -- ← 必加硬上限
)
```

**历史教训**: CLAUDE.md 2026-07-12 提到 PG recursive CTE 循环引用爆内存事故, 必加
`depth < N` 上限。

### 铁律 3: 跨文件 dry-run 默认 True

service + Celery task + CLI 三层入口都默认 `dry_run=True`, 显式
`--apply` / `dry_run=False` 才写库。防误操作是 PR14 的核心 UX 设计。

| 入口 | dry-run 默认 | 真写库参数 |
| - | - | - |
| `service.backfill_all_paths()` | `dry_run=True` | `dry_run=False` |
| `service.backfill_for_file()` | `dry_run=True` | `dry_run=False` |
| `service.backfill_for_folder()` | `dry_run=True` | `dry_run=False` |
| `task.backfill_paths_task()` (Celery) | `dry_run=False` (后台) | `dry_run=True` (看效果) |
| `scripts/backfill_drive_comments_path.py` | (无, 默认 dry) | `--apply` |

**设计取舍**: Celery task 默认 `dry_run=False` (因为是后台慢速回填, 需要真写库);
但调用方传 `dry_run=True` 仍可走 dry-run 模式看效果。

### 铁律 4: Celery 慢速回填用 NullPool

复用 chat_history_tasks 模式
(`create_celery_engine_and_session + NullPool + asyncio.run + engine.dispose`),
避免连接池跨事件循环 (`Future attached to different loop`)。

```python
# 正模式
async def _run():
    engine, session_factory = create_celery_engine_and_session()
    try:
        async with session_factory() as db:
            # ... 业务逻辑
    finally:
        await engine.dispose()

result = asyncio.run(_run())

# 反模式 (绑主进程 loop, Celery 跨 task 爆)
async with AsyncSessionLocal() as db:  # AsyncSessionLocal 是全局, 绑主进程 loop
    ...
```

**CLAUDE.md 2026-06-03 教训复用**: drive_cleanup / chat_history / drive_collab
都走这个模式, PR14 drive_comments_path_backfill 继续沿用。
