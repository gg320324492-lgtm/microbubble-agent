## 5 新铁律 (W68 第 8 批 A-1 沉淀)

### 铁律 1: 合并前必 fetch 验证 tip commit

```bash
git fetch origin <branch>
git log origin/<branch> --oneline -3
```

**理由**: origin 可能有多个 commit (不只是分支 HEAD), 派工的 agent 在 merge 期间可能 push 新修复. fetch 后用 `git log origin/<branch> -3` 看最新 3 commits, 避免漏 commit 或 merge 错 commit.

### 铁律 2: alembic 串单链纪律 (CLAUDE.md 已沉淀, 本任务强化)

```bash
python -c "from alembic.script import ScriptDirectory; ..."
# 期望: Count: 1 (单 head)
```

**理由**: 多分支并行派 alembic migration 是高危操作. W68 第 3 批 062/063 双头事故已修, 但 W68 第 5 批 + 第 7 批新增 064/065 必须串单链. 064 接 063, 065 接 064, **链结构**: 061 → 062 → 063 → 064 → 065. 任何 alembic commit 合并后必须验证 single head.

**应用**: 本任务合并 `feat/mobile-v3.2-push-backend-2026-07-24` 后, 验证 `065_push_subscriptions` 是唯一 head. PASS.

### 铁律 3: 冲突时优先看 branch commit message 决定哪边更新

不要默认 `git checkout --ours` 或 `--theirs`. 决策顺序:
1. 看分支 commit message + 路线 (路线 A/B/C/D/E/F/G/H) + batch 编号 (第 5/6/7 批)
2. 一般规则: **新批次 > 老批次**, 因为新批次的 narrative 应覆盖老批次
3. 例外: add/add 冲突两个 branch 互斥, 选**更完整更长**的版本
4. 代码冲突 (如 main.py 新增 router): 两个都保留 (除非明显重复)

### 铁律 4: push 前必看 head commit 是否与本地一致

```bash
git rev-parse HEAD
git ls-remote origin main
# 两个 hash 必须一致
```

**理由**: 防止 push 失败后不知道 origin 实际状态, 避免"以为 push 了实际没 push"的事故. 本任务中:
- 本地 HEAD: `2a468211ce5ab9a81ae6f6f9850909b9ea15fc88`
- 推送后 `git ls-remote origin main` 返回: `2a468211ce5ab9a81ae6f6f9850909b9ea15fc88`
- 完全一致, push 成功
