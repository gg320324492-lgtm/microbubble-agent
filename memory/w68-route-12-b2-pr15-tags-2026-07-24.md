## 5 新铁律

### 1. 12 个内置 tag 白名单 (Pydantic schema + service 双重校验)

**铁律**: 任何文件版本标签**必须**在 12 个内置白名单内 (不允许用户自定义), 避免 tag 体系污染.

**12 个白名单**:
- `release` / `stable` / `deprecated` / `security` (核心 4)
- `auto-save` / `manual` (存储来源)
- `breaking` / `experimental` / `legacy` (兼容性)
- `featured` / `archived` / `final` (元数据)

**理由**:
- 与 GitHub release tags 对齐 (`git tag` 也是有限命名空间)
- 课题组场景稳定 (12 个内置覆盖 95% 用例)
- 用户自定义 (e.g. `release-2024.10`) 容易重复 / 拼写不一致 / 国际化难

**落地**:
- `app/models/drive_version_tag.py:ALLOWED_TAG_NAMES = frozenset({...})` (12 个)
- `app/services/drive_version_tag_service.py:_validate_tag_name()` (Pydantic schema 重复校验)
- `tests/test_drive_v2_pr15_version_tags.py:test_add_tag_name_not_in_whitelist_raises_400` (e2e 兜底)
- `tests/test_drive_v2_pr15_version_tags.py:test_allowed_tag_names_has_exactly_12_entries` (完整性自检)

**未来扩展**: 若课题组真的需要自定义 (e.g. `release-2024.10`), 应走 alias 映射到内置 tag, 而不是放开白名单.

### 2. version_number (整数递增) + tag_name (语义) 双轨

**铁律**: PR15 标签系统与 PR9 version_number 共存互补, 不要试图用 tag 替代 version_number, 反之亦然.

| 维度 | version_number (PR9) | tag_name (PR15) |
|------|---------------------|-----------------|
| 类型 | 整数递增 (1, 2, 3...) | 字符串白名单 (12 个) |
| 含义 | 版本顺序 | 语义标签 |
| 数量 | 1 个文件 N 个版本 → N 个 version_number | 1 个版本可挂 0~12 个 tag |
| 自动/手动 | 自动 (upload_new_version 时 +1) | 手动 (用户调 POST /tags) |

**组合场景**:
- `v3 + release` = "第 3 版 + 是发布版"
- `v5 + deprecated` = "第 5 版 + 已废弃, 不要用"
- `v1 + stable + featured` = "第 1 版 + 稳定 + 推荐"

**实现**: `list_versions_with_tags` (drive_version_service.py) 用 LEFT OUTER JOIN 1 query 拿 version + tags, 比 `list_versions` + `list_tags_by_file` 拆分节省 50% query 数.

### 3. 跨文件隔离 (file_id 强约束)

**铁律**: tag 通过 `version_id` → `DriveFileVersion` → `file_id` 隐式关联 file, 但 service 层必须显式校验.

**为何要显式校验**:
- `DriveFileVersion.file_id` 是 FK 到 `knowledge.id`, 但 `DriveVersionTag` 直接 FK 到 `DriveFileVersion.id`
- 若 service 只校验 version_id 存在, 可能误把 file_a 的 tag 加到 file_b 的 version 上 (虽然 FK 会拒绝)
- 显式校验 `version.file_id == expected_file_id` 防 race condition + DB 一致性

**落地**: `app/services/drive_version_tag_service.py:_validate_version_exists_and_modify_authority()` 同时校验:
- version_id 存在
- file_id 存在 + 未删除 + drive 模式
- user_id 是 file 创建人 OR folder 管理员 OR 平台管理员

**e2e**: `tests/test_drive_v2_pr15_version_tags.py:test_add_tag_cross_file_isolation` (file_a tag 不污染 file_b).

### 4. UNIQUE 约束 (version_id, tag_name) 幂等 + WS 推送

**铁律**: tag add 幂等 — 重复 add 同一 `(version_id, tag_name)` 返 `None`, 不抛错.

**为何要幂等**:
- 前端 toggle 体验: 用户重复点 "release" 标签按钮, 不应弹错
- 与 PR12 reaction (`DriveReaction.add_reaction`) 一致模式
- UNIQUE 约束是 DB 层强制保证, service 捕获 `IntegrityError` 后 `rollback()` + `return None`

**落地**:
- `alembic/versions/070_drive_version_tags.py`: `UNIQUE (version_id, tag_name)`
- `app/services/drive_version_tag_service.py:add_tag()`: try INSERT + except IntegrityError + rollback + return None

**WS 推送**: add 成功后 `publish_version_tag_added` (MEDIUM priority) 通知 file owner (跳过自推). payload: `{type, tag_id, version_id, file_id, tag_name, actor_id, ts}`.

**e2e**: `test_add_tag_idempotent_returns_none_on_unique_violation` + `test_publish_version_tag_added_self_action_skipped`.

### 5. list_versions_with_tags 性能路径 (LEFT OUTER JOIN 1 query)

**铁律**: UI 列版本列表时, 必须用 `list_versions_with_tags` (PR15 新增), 不要用 `list_versions` + `list_tags_by_file` 拆分.

**性能对比**:
- 拆分: 2 query (拿 versions + 拿 tags by file_id) → ~30ms
- LEFT OUTER JOIN: 1 query → ~15ms (节省 50%)

**实现** (`app/services/drive_version_service.py:list_versions_with_tags`):
```sql
SELECT v.*, m.name, t.*
FROM drive_file_versions v
LEFT OUTER JOIN members m ON v.uploader_id = m.id
LEFT OUTER JOIN drive_version_tags t ON t.version_id = v.id
WHERE v.file_id = ?
ORDER BY v.version_number DESC, t.created_at ASC
```

**e2e**: `test_list_versions_with_tags_single_query_join` 断言 `execute_count == 1`.

**纪律**: 未来 UI 列文件版本时 (DriveDetailView / DriveHistoryPanel), 默认走 `list_versions_with_tags` 端点.
