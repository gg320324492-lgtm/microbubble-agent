## 3. 结论

### 3.1 子 plan ②

七维评分是已实施而非待开发，真实路径是 `seven_d_scoring.py`，应做权重/报告/780 抽样验收。save_to_kb 已具备灰度、长度参数、observer 和 rollback 关联，但 dedup、LLM 拒答、敏感词、5% 抽检尚未形成原计划的独立五道防线证据，应标“部分”。自动回滚目前更接近 `scripts/auto_intake_rollback.py` + API/JSON 监控，而不是计划中的 Celery service task，worker 注册和幂等性需 W71 补证。KB automation 和 Dashboard 有历史提交与产物，但闭环 E2E、四卡片接口、5 分钟刷新和负反馈暂停仍需验收。smoke 200 workflow 已有；本地 baseline 因 Redis 未启动为 69 PASS + 7 SKIP + 2 connection failures，不能宣布绿，CI 依赖栈中必须重跑。

### 3.2 子 plan ③

NavRail、ThinkingModeSwitch、ChatBreadcrumb 均已在 `web/src/components/chat/` 存在，ChatViewSSE 已集成。Mobile TabBar/Drawer 覆盖了移动导航的一部分，但不等同于 desktop/mobile 完整统一 NavRail。思考模式三档和面包屑需做 store 映射、a11y、长标题、移动视口及视觉 baseline 验收。因此 UI 不是从零实施，W72 应派 3 agents 做验收与小修，不应照原四 agent 计划重复创建。

## 4. 主拍方案

| 选项 | W71 范围 | W72 | 判断 |
|---|---|---|---|
| **A（推荐）** | 5 agents：评分、五防线、rollback、KB E2E、Dashboard+CI | 3 agents 收 UI | 先质量和 intake 风险，最稳 |
| B | 8 agents，②③并行 | 无续批 | 周期短但冲突/重复风险高 |
| C | 4 agents，跳过 KB 闭环 | 续派 UI/KB | 仅在 intake 继续关闭时可用 |
| D | W71 不派 plans，商业化优先 | W72+再排 | partial 和 baseline 红灯继续漂移 |

推荐 A。保持 `AUTO_KB_INTAKE_ENABLED` 默认关闭；先解决依赖环境和质量门禁，再考虑灰度开放。W71 DoD 是五防线逐项有审计/负测、回滚可发现且幂等、KB 有 pass/reject/rollback/negative-feedback 四路径、smoke 200 在完整依赖环境通过。W72 DoD 是三组件双栈/a11y/视觉回归验收。
